## 正文
<h2 align="center">DevOps</h2>
<p>
在游戏开发过程中，要不断地出数据和产品的版本来进行测试、调试和发布，这些过程如果全部依赖人工操作是非常低效且容易出错的。
通过建设 DevOps 体系（CI/CD、版本生产线、工程管理等），可以用自动化工具链和规范化流程，大幅提升项目的生产效率和稳定性。
</p>

**关键词:**<br/>
*CI,CD,DevOps,Maven,Jenkins,n8n,出包生产线,工程管理,AI Coding*

**标签:**<br/>
*等级: 中级|高级, 阶段: 开发, 分类: 生产能力, 角色: 客户端开发|服务端开发|运维*

## 持续集成（CI）

| 维度 | 内容 |
|------|------|
| **作用** | 最大化地实现生产自动化，降低人力投入，让每次提交都尽快得到「可运行、可测试」的反馈 |
| **应用场景** | 自动化构建、自动化测试、自动化发布、提高开发效率 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现持续集成** 需要实现自动化生产流程 | 设计 CI 流程；实现自动化工具；优化 CI 性能 |

| 应用举例 | 说明 |
|----------|------|
| **游戏包自动构建** | 实时为 QA 提供可测试版本；及时准备可发布版本 |
| **自动测试** | 尽快发现和定位问题；单元测试、性能测试等；每次提交自动运行测试 |
| **数据发布自动化** | 自动提交、发布配置、活动表、资源清单等 |
| **运维辅助** | 批量服务器管理（更新、停服、开服、重启、检查等） |

| 要点和思考方向 |
|----------------|
| 持续集成是提高开发效率的重要手段 |
| 实现自动化减少人力投入 |
| 快速反馈提高问题发现速度 |
| 持续优化 CI 流程提高效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「自动构建」「自动测试」「数据发布」「运维」「Jenkins」 |
| **方法** | 让 AI 设计 CI 流程、触发条件、工具串联 |
| **应用** | 游戏包构建、自动测试、数据发布、运维脚本 |
| **提示词范例** | 「游戏项目需 CI：代码提交触发自动构建、自动测试、配置/活动表发布，请设计 Jenkins 流水线及与 Unity/服务端的集成步骤」 |

| 选择合适的工具提高效率 |

## 版本交付生产线

| 维度 | 内容 |
|------|------|
| **作用** | 搭建一条「可以实现批量快速出包」的统一生产线 |
| **应用场景** | 批量出包、多渠道支持、多平台支持、提高出包效率 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决多渠道多平台问题** 渠道多、平台多，对接耗时耗力 | 产品出包：统一出包流程、多渠道支持；对接平台（账户、支付）SDK；实现批量、快速处理 |

| **应用渠道差异信息** | 渠道差异应用 |
| **构建安装包** | 安装包构建 |
| **输出到指定位置** | 本地或制品仓库 |

| 要点和思考方向 |
|----------------|
| 版本交付生产线是批量出包的基础 |
| 解决多渠道多平台问题 |
| 实现批量快速出包 |
| 持续优化生产线流程 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「出包」「多渠道」「多平台」「资源」「Bundle」「渠道差异」 |
| **方法** | 让 AI 设计生产线流程、步骤串联、渠道差异化处理 |
| **应用** | 出包流水线、多渠道支持、资源构建 |
| **提示词范例** | 「游戏需出包生产线：更新资源数据→构建资源→生成 Bundle→生成清单→应用渠道差异→构建安装包→输出，请设计脚本流程及多渠道差异化配置方案」 |

## 工作流自动化工具

| 维度 | 内容 |
|------|------|
| **作用** | 打通各类服务与工具，通过可视化工作流串联通知、数据同步、触发等自动化流程 |
| **应用场景** | 构建完成通知、测试结果推送、运维告警、多系统数据同步、DevOps 工具链串联 |

| 问题 | 解决方向 |
|------|----------|
| **如何串联 DevOps 工具链** 构建、测试、部署等环节需与通知、IM、工单等打通 | 使用工作流自动化工具连接 Jenkins、Git、Slack/钉钉、邮件等；设计触发条件与数据流转 |
| **如何实现自托管与可控** 部分团队希望工作流数据不出内网 | 选择支持自托管、可私有部署的工具 |

| 候选 | 特点 |
|------|------|
| **n8n** | 开源工作流自动化平台；支持自托管（Docker/npm）；可视化编排；400+ 集成；Webhook/Cron 触发；可扩展自定义节点 |
| **Zapier** | 云服务；丰富集成；按量付费 |
| **Power Automate** | 微软平台；Office 365 集成；企业级 |

| 典型应用 | 说明 |
|----------|------|
| **构建完成通知** | Jenkins 构建结束 → 工作流 → 推送结果到钉钉/Slack/邮件，附带构建链接与失败原因 |
| **测试结果汇总** | 测试报告生成 → 自动汇总并推送到指定群组 |
| **运维告警** | 服务器/服务异常 → 触发工作流 → 通知值班人员、创建工单 |

| 要点和思考方向 |
|----------------|
| 工作流自动化是 DevOps 工具链串联的补充手段 |
| 自托管方案（如 n8n）适合对数据安全有要求的团队 |
| 与 CI 工具（Jenkins 等）配合，实现构建→通知的闭环 |
