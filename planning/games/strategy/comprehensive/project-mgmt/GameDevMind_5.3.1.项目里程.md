## 常见问题

## 网游 vs 主机

| 问题 | 解答 |
|------|------|
| 为什么主机游戏和PC网游、手机网游的里程碑不太一样？ | 因为网游可以在游戏上线后持续的更新、维护，但主机游戏的审查机制和迭代更新方式都不同，所以产品需要在发布之前尽量做的完善。 |

| 类型 | 特点 |
|------|------|
| **网游** | 可以持续迭代，上线后可以修复问题、添加内容，里程碑更注重快速验证和迭代 |
| **主机游戏** | 发布后更新困难，需要通过严格的审查，里程碑更注重完整性和稳定性 |

## 自研 vs 外包

| 问题 | 解答 |
|------|------|
| 自研和外包有什么不同？ | 自研以产品和盈利为中心；外包以时间和质量为中心。 |

| 模式 | 特点 |
|------|------|
| **自研** | 以产品和盈利为中心：关注产品的市场表现和盈利能力；在有限成本内做出吸引人的产品：需要在成本控制下创造有竞争力的产品 |
| **外包** | 以时间和质量为中心：关注按时交付和交付质量；按时、按质的去完成客户的需求：严格按照合同要求完成开发任务 |

## 常规里程

## 立项

| 维度 | 内容 |
|------|------|
| **作用** | 项目启动阶段，确定项目方向和目标 |
| **应用场景** | 所有游戏项目的起始阶段；新项目启动；项目方向调整 |
| **做什么的？** | 项目启动阶段，确定项目方向和目标 |
| **在哪用？** | 所有游戏项目的起始阶段 |

| 问题 | 解决方向 |
|------|----------|
| **如何确定项目方向？** 周期最短，意义最重大的一个阶段，决定了整个项目的方向 | 头脑风暴集思广益、探索可能性；确定团队要做什么（创意导向：以创意为核心、探索新市场；市场导向：以市场为导向、满足市场需求）；进行市场调研；分析竞品；评估团队能力；确定项目定位 |
| **如何评估项目的可行性？** 确定团队能做什么，评估技术能力和资源，确保项目可执行 | 评估技术可行性；评估资源可行性；评估市场可行性；评估团队能力匹配度 |
| **如何平衡创意和市场？** | 明确项目定位（创意导向 vs 市场导向）；在创意和市场之间找到平衡点；考虑目标用户需求；评估市场接受度；制定差异化策略 |

| 要点和思考方向 |
|----------------|
| 立项是项目的基础，决定了整个项目的方向 |
| 周期最短但意义最重大，需要充分思考和讨论 |
| 通过头脑风暴探索各种可能性 |
| 明确团队要做什么（创意导向 vs 市场导向） |
| 评估团队能做什么，确保项目可执行 |
| 平衡创意和市场，找到合适的定位 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「立项」「头脑风暴」「创意导向」「市场导向」「可行性」 |
| **方法** | 让 AI 协助头脑风暴、市场分析、可行性评估 |
| **应用** | 立项文档、市场调研、竞品分析 |
| **提示词范例** | 「游戏项目立项，需确定方向（创意/市场导向），请协助设计头脑风暴提纲、可行性评估维度，以及竞品分析框架」 |

## 设计

| 维度 | 内容 |
|------|------|
| **作用** | 将想法具体化，确定产品的外观、框架和技术方案 |
| **应用场景** | 立项后的设计阶段；产品概念设计；技术方案设计；视觉风格设计 |
| **做什么的？** | 将想法具体化，确定产品的外观、框架和技术方案 |
| **在哪用？** | 立项后的设计阶段 |

| 问题 | 解决方向 |
|------|----------|
| **如何将抽象的想法转化为具体的设计？** 原画：视觉化验证想法可行性；统一团队视觉认知；从画面中发现新问题或机会 | 制作原画和概念图；制作设计文档；进行设计评审 |
| **如何确保团队对产品有统一的理解？** | 通过原画和视觉设计统一认知；制作详细的设计文档；进行设计评审和讨论；建立设计规范；定期沟通和同步 |
| **如何选择合适的技术方案？** 技术选型：确定设计框架（设计模式、架构）；确定技术选型（技术栈、工具） | 评估技术可行性；考虑团队技术能力；考虑项目需求；评估技术成本和风险 |

| 要点和思考方向 |
|----------------|
| 设计是将想法具体化的关键阶段 |
| 通过原画和视觉设计验证想法可行性，统一团队认知 |
| 从视觉化过程中发现新的问题或机会 |
| 确定产品的设计框架和技术选型 |
| 确保团队对产品有统一的理解 |
| 选择合适的技术方案平衡需求和能力 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「设计」「原画」「技术选型」「架构」「设计文档」 |
| **方法** | 让 AI 协助设计文档结构、技术选型、架构设计 |
| **应用** | 设计文档、技术方案、视觉规范 |
| **提示词范例** | 「设计阶段需将想法具体化，请协助设计产品设计文档结构、技术选型评估表，以及原画与概念图的需求说明模板」 |

## 原型/First Playable

| 维度 | 内容 |
|------|------|
| **作用** | 快速制作可体验的核心玩法版本，验证产品概念 |
| **应用场景** | 设计阶段之后，正式开发之前；需要验证核心玩法时；需要验证产品概念时 |
| **做什么的？** | 快速制作可体验的核心玩法版本，验证产品概念 |
| **在哪用？** | 设计阶段之后，正式开发之前 |

| 问题 | 解决方向 |
|------|----------|
| **如何快速验证核心玩法？** 快速做出来可体验版本，验证核心内容、玩法、产品感觉 | 聚焦核心玩法，简化其他功能；使用快速原型工具；快速迭代和测试 |
| **如何控制原型的成本？** | 聚焦核心功能，避免过度开发；使用现有资源和工具；快速迭代，及时调整；控制开发时间和资源投入 |
| **何时应该调整方向？** 在投入大量资源之前，及时调整方向成本较低 | 根据验证结果判断；发现核心玩法不可行时及时调整；发现市场反馈不佳时考虑调整；在投入大量资源前调整 |

| 要点和思考方向 |
|----------------|
| 原型是验证产品概念的关键阶段 |
| 快速制作可体验版本验证核心玩法和产品感觉 |
| 聚焦核心功能，控制原型成本 |
| 在投入大量资源前及时调整方向，成本较低 |
| 通过实际体验验证想法的可行性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「原型」「FPP」「核心玩法」「验证」「快速迭代」 |
| **方法** | 让 AI 协助设计原型范围、验证清单、成本控制 |
| **应用** | 原型范围、验证标准、调整决策 |
| **提示词范例** | 「需制作 First Playable 验证核心玩法，请协助设计原型范围（必须/可省略）、验证清单，以及何时应调整方向的判断标准」 |

## 计划/Planning

| 维度 | 内容 |
|------|------|
| **作用** | 制定详细的开发计划和时间表 |
| **应用场景** | 原型验证之后，正式开发之前；需要制定详细开发计划时；需要明确开发时间表时 |
| **做什么的？** | 制定详细的开发计划和时间表 |
| **在哪用？** | 原型验证之后，正式开发之前 |

| 问题 | 解决方向 |
|------|----------|
| **何时制定详细计划？** 建议原型验证后制定，此时理解更深入、计划更准确；不同团队策略不同，取决于领导者风格和项目特点 | 根据项目特点选择时机；在原型验证后制定更准确；根据团队风格调整 |
| **如何平衡计划的详细程度和灵活性？** | 核心功能制定详细计划；非核心功能保持灵活性；短期计划详细，长期计划灵活；建立计划调整机制；定期回顾和调整计划 |
| **如何应对计划的变化？** | 建立计划变更流程；保持计划的灵活性；及时沟通和同步变化；评估变化的影响；调整资源和时间安排 |

| 要点和思考方向 |
|----------------|
| 计划是项目执行的基础 |
| 建议在原型验证后制定详细计划，此时理解更深入 |
| 不同团队有不同的策略，取决于领导者风格和项目特点 |
| 平衡计划的详细程度和灵活性 |
| 建立计划变更机制应对变化 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「计划」「Planning」「里程碑」「甘特图」「变更」 |
| **方法** | 让 AI 协助制定计划模板、里程碑定义、变更流程 |
| **应用** | 开发计划、时间表、变更管理 |
| **提示词范例** | 「原型验证后需制定详细计划，请协助设计开发计划模板、核心/非核心功能的计划粒度，以及计划变更的评估与沟通流程」 |

## 内测

| 维度 | 内容 |
|------|------|
| **作用** | 邀请部分玩家进行多方面的测试，确保游戏质量 |
| **应用场景** | 封测之后，公测之前；需要全面测试游戏时；需要验证游戏质量时 |
| **做什么的？** | 邀请部分玩家进行多方面的测试，确保游戏质量 |
| **在哪用？** | 封测之后，公测之前 |

| 问题 | 解决方向 |
|------|----------|
| **如何组织内测？** 全面测试运行性能、游戏设计、平衡性、BUG、服务器负载等，确保公测顺利进行 | 确定内测规模和范围；选择测试玩家；准备测试环境；建立测试流程 |
| **如何处理内测数据？** 内测资料进入公测通常不保留，可采取公测奖励措施或不删档内测 | 根据策略决定数据保留方式；考虑玩家体验和游戏公平性；制定内测奖励机制 |

| 要点和思考方向 |
|----------------|
| 内测是全面测试游戏质量的重要阶段 |
| 对游戏运行性能、设计、平衡性、BUG、服务器负载等多方面测试 |
| 确保公测能顺利进行 |
| 根据策略决定是否保留内测数据 |
| 平衡玩家体验和游戏公平性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「内测」「性能」「平衡性」「服务器」「数据保留」 |
| **方法** | 让 AI 协助设计内测方案、测试维度、数据策略 |
| **应用** | 内测方案、测试清单、数据策略 |
| **提示词范例** | 「内测需全面测试性能、设计、平衡性、BUG、服务器负载，请协助设计内测测试清单、数据保留策略（删档/不删档），以及公测衔接要点」 |

|------|----------|

| **如何处理公测数据？** 根据策略决定数据保留方式，平衡玩家体验和游戏公平性 | 根据策略决定数据保留方式；考虑玩家体验和游戏公平性；制定公测奖励机制 |

| 要点和思考方向 |
|----------------|
| 公测是面向所有玩家的大规模测试阶段 |
| 验证游戏在大规模用户下的稳定性 |
| 根据策略决定数据保留方式，平衡玩家体验和游戏公平性 |
| 准备应对大规模用户和问题 |

| 类型 | AI Coding 指南 |
|------|----------------|
