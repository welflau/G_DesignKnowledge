## 游戏分类

| 维度 | 内容 |
|------|------|
| **作用** | 不同游戏类型需要不同的业务功能 |
| **应用场景** | 根据游戏类型设计相应的业务功能、游戏架构设计、功能需求分析 |

| 问题 | 解决方向 |
|------|----------|
| **如何根据游戏类型设计业务功能** 不同游戏类型需要不同的业务功能 | 分析游戏类型特点；设计匹配的业务功能；实现功能模块 |

| 游戏类型 | 需要的业务功能 |
|----------|----------------|
| **SLG（策略游戏）** | 资源管理、联盟系统、战争系统等 |
| **Racing（赛车游戏）** | 竞速系统、车辆系统、赛道系统等 |
| **FPS（第一人称射击）** | 匹配系统、武器系统、地图系统等 |
| **Puzzle（益智游戏）** | 关卡系统、评分系统、道具系统等 |
| **MMORPG（大型多人在线角色扮演）** | 用户系统、聊天系统、任务系统、好友系统、公会系统等 |
| **CCG/TCG（集换式卡牌游戏）** | 卡牌系统、对战系统、收集系统等 |
| **RPG（角色扮演游戏）** | 角色系统、任务系统、装备系统、技能系统等 |
| **Action（动作游戏）** | 战斗系统、连击系统、技能系统等 |
| **Sandbox（沙盒游戏）** | 建造系统、创造系统、探索系统等 |
| **RTS（即时战略游戏）** | 资源系统、单位系统、建筑系统等 |
| **Sports（体育游戏）** | 比赛系统、球员系统、训练系统等 |

| 要点和思考方向 |
|----------------|
| 根据游戏类型选择合适的业务功能 |
| 避免过度设计，只实现必要的功能 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明游戏类型；如「MMORPG」「MOBA」「SLG」「RPG」 |
| **方法** | 让 AI 根据游戏类型推荐业务功能、模块划分 |
| **应用** | 功能规划、架构设计、需求分析 |
| **提示词范例** | 「做 MMORPG 手游，请列出核心业务功能模块（用户、聊天、任务、好友、公会等），以及各模块的优先级和依赖关系」 |

|------|----------|
| **如何管理登录状态和流程** 需要管理用户登录状态和流程 | 登录状态与流程：登录状态机、登录流程、状态持久化；异常处理：网络异常、账号异常、重连机制；多种登录方式；token 管理；会话管理；安全验证 |

| **数据验证和安全** 需要验证数据和安全保护 | 输入验证；数据加密；防刷机制；安全日志 |

| 模块 | 状态与流程 | 异常处理 |
|------|------------|----------|

| **支付** | 支付状态机、支付流程、订单管理 | 支付失败、超时、回调处理 |

| 要点和思考方向 |
|----------------|
| 登录：状态与流程、异常处理 |
| 支付：状态与流程、异常处理 |
| 数据验证 |
| 登录和支付流程要稳定可靠 |
| 实现完善的异常处理机制 |
| 保证数据安全和用户隐私 |

| 类型 | AI Coding 指南 |
|------|----------------|

| **方法** | 让 AI 设计登录流程、支付状态机、异常与重试 |
| **应用** | 登录系统设计、支付对接、状态机实现 |
| **提示词范例** | 「用户系统需登录状态机：未登录、登录中、已登录、登出中，请设计 C# 状态枚举、转换规则，以及支付回调的幂等与重试处理」 |

## 聊天系统

| 维度 | 内容 |
|------|------|
| **作用** | 玩家之间的交流系统 |
| **应用场景** | 需要玩家交流的游戏、社交功能、游戏内沟通 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现关键词过滤** 需要过滤不当言论 | 实现关键词库、过滤算法、替换机制、自定义关键词；使用敏感词库；实时过滤；过滤日志 |
| **如何防止刷屏** 需要防止玩家刷屏 | 发言频率限制；冷却时间；刷屏检测；自动禁言；频率控制；异常检测；惩罚机制 |
| **如何实现同步刷新** 需要同步聊天消息 | 消息同步、推送、缓存；使用消息队列；实时推送；优化网络传输 |
| **聊天功能实现** 需要实现聊天功能 | 频道：点对点（私聊）、广播（世界、公会、队伍）；禁言；历史聊天数据（通常不长期保存） |

| 功能 | 说明 |
|------|------|
| **频道** | 点对点（私聊通道、消息加密、离线消息）；广播（世界频道、公会频道、队伍频道） |
| **禁言** | 禁言管理、禁言时间、禁言通知 |
| **历史聊天数据** | 通常游戏不会长期保存，实现消息缓存、清理、查询 |

| 要点和思考方向 |
|----------------|
| 功能：频道（点对点、广播）、禁言、历史聊天数据 |
| 解决问题：关键词过滤、防止快速发言刷屏、有同步刷新机制 |
| 聊天系统要保证消息及时同步 |
| 实现完善的过滤和管理机制 |
| 优化性能支持大量并发消息 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「频道」「敏感词」「刷屏」「消息同步」 |
| **方法** | 让 AI 设计频道结构、敏感词过滤、频率限制、消息推送 |
| **应用** | 聊天架构、敏感词过滤、防刷屏 |
| **提示词范例** | 「聊天系统需世界频道、公会频道、私聊，请设计消息格式、敏感词过滤算法（DFA 或 Trie），以及发言频率限制的防刷逻辑」 |

## 匹配系统

| 维度 | 内容 |
|------|------|
| **作用** | 将玩家匹配到一起进行游戏 |
| **应用场景** | 需要匹配玩家的游戏（如 MOBA、FPS）、多人对战游戏、竞技游戏 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理海量用户的匹配** 需要处理大量用户的匹配请求 | 匹配队列；分组匹配；房间分组；房间分配；匹配算法（网络延迟与竞争性平衡）；分布式匹配服务；负载均衡 |
| **如何平衡匹配速度和匹配质量** | 匹配策略配置；动态调整；匹配监控；匹配优化 |
| **第三方匹配服务** 可以使用第三方匹配服务 | 候选：亚马逊 GameLift；评估第三方服务；实现服务对接与监控 |

| 方案 | 说明 |
|------|------|
| **玩家请求匹配后进入队列等待** | 匹配队列、队列管理、队列优化 |
| **建立多个匹配队列分组，排队的玩家进行分组排队** | 分组策略、分组管理、分组优化 |
| **按匹配条件为房间创建分组** | 房间管理、房间分组、房间分配 |
| **基于条件轮训房间，分配玩家到房间** | 分配算法、分配优化、分配监控 |
| **玩家匹配逻辑：在网络延迟和竞争性之间做出最佳平衡** | 匹配算法、延迟优化、平衡性优化 |

| 候选第三方 | 功能 | 特点 | 适用 |
|------------|------|------|------|
| **亚马逊 GameLift** | 匹配、服务器托管 | 专业、稳定 | 大型游戏 |

| 要点和思考方向 |
|----------------|
| 解决问题：海量用户的匹配 |
| 匹配系统要平衡速度和质量 |
| 实现高效的匹配算法 |
| 支持大量并发匹配请求 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「匹配队列」「MMR」「延迟」「房间分配」 |
| **方法** | 让 AI 设计匹配队列、分组策略、房间分配、匹配算法 |
| **应用** | 匹配系统设计、匹配算法、队列管理 |
| **提示词范例** | 「MOBA 匹配需 5v5，根据 MMR 和等待时间匹配，请设计匹配队列、分组策略、房间分配流程，以及网络延迟与竞争性的平衡考虑」 |

## 排行系统

| 维度 | 内容 |
|------|------|
| **作用** | 显示玩家的排名 |
| **应用场景** | 需要排行榜的游戏、竞技游戏、社交游戏 |

| 问题 | 解决方向 |
|------|----------|
| **排行数据的刷新时机如何控制** 需要控制排行数据的刷新时机 | 实时刷新、定时刷新、增量刷新；实现刷新策略、优化、监控 |
| **排行榜性能优化** 排行榜可能影响性能 | 排行榜缓存；分页加载；异步刷新；优化查询性能 |
| **排行榜功能扩展** 需要扩展排行榜功能 | 多种排行榜类型；排行榜分组；排行榜奖励；排行榜历史 |

| 刷新方式 | 优点 | 缺点 | 适用 |
|----------|------|------|------|
| **实时刷新** | 数据准确 | 性能开销大 | 小规模排行榜 |
| **定时刷新** | 性能开销小 | 数据可能延迟 | 大规模排行榜 |
| **增量刷新** | 平衡性能和准确性 | 实现复杂 | 中等规模排行榜 |

| 要点和思考方向 |
|----------------|
| 解决问题：排行数据的刷新时机 |
| 排行榜刷新要平衡性能和准确性 |
| 实现高效的排行榜查询 |
| 支持多种排行榜类型 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「排行榜」「实时」「定时」「刷新」「分页」 |
| **方法** | 让 AI 设计刷新策略、缓存、分页、Redis 实现 |
| **应用** | 排行榜设计、刷新策略、性能优化 |
| **提示词范例** | 「排行榜日榜需定时刷新、支持分页查询，请设计 Redis 有序集合的存储结构、定时刷新逻辑，以及分页查询的 C# 或 Go 实现」 |
