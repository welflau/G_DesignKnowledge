## 代码重构概述

| 维度 | 内容 |
|------|------|
| **作用** | 在不改变外部行为的前提下，改善内部结构，提高质量 |
| **应用场景** | 难以维护/理解/扩展；消除技术债务；提高可读性；优化性能；添加新功能前整理 |

**会遇到哪些问题？用什么解决？**

| 问题 | 解决方向 |
|------|----------|
| **何时需要重构** | 代码坏味道；添加功能困难；难以理解；测试难写；性能问题 |
| **安全重构** | 完善测试；小步重构；版本控制；先理解再重构；IDE 重构工具 |
| **避免引入新 bug** | 重构前有测试覆盖；重构后跑测试；自动化验证；代码审查；逐步验证 |
| **重构时机** | 添加功能时顺便重构；定期技术债务日；平衡重构与功能；持续过程 |

| 要点和思考方向 |
|----------------|
| 持续改进；必须有测试；小步重构；只改结构不改功能 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明重构目标、风险点、测试覆盖；如「抽 Controller 逻辑」「无单元测试」「需保持行为不变」 |
| **方法** | 要求先补充测试；小步提交；提供重构前后对比；说明行为不变 |
| **应用** | 制定重构计划；生成测试用例；分步重构指导 |
| **提示词范例** | 「这段 200 行的 Update 逻辑要重构，目前无单元测试，请先给出关键路径的测试用例，再分步提取函数」 |

## 提炼函数（Extract Function）

| 维度 | 内容 |
|------|------|
| **作用** | 将长函数拆成小函数，提高可读性和可维护性 |
| **应用场景** | 函数过长（>20–30 行）；做多件事；有重复代码块 |

| 问题 | 解决方向 |
|------|----------|
| **函数粒度** | 只做一件事；函数名清楚；避免过度拆分；单一职责 |

| 要点和思考方向 |
|----------------|
| 长函数应拆分；函数名表达意图；小函数易测试 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明函数长度、职责数量、重复块；如「80 行函数做 3 件事」「有 3 处相似逻辑」 |
| **方法** | 要求单一职责；函数名即文档；避免过度拆分；提供参数建议 |
| **应用** | 提取函数；命名建议；识别可提取块 |
| **提示词范例** | 「将这段 60 行的初始化函数拆成小函数，每个函数名表达意图，保持单一职责」 |

## 降低耦合

| 维度 | 内容 |
|------|------|
| **作用** | 减少模块间依赖，提高模块化 |
| **应用场景** | 依赖过多；修改影响多模块；难以单独测试 |

| 问题 | 解决方向 |
|------|----------|
| **依赖关系复杂** | 依赖注入；接口抽象；事件/消息；依赖倒置；观察者模式 |

| 要点和思考方向 |
|----------------|
| 高耦合难维护；用接口和抽象；依赖注入有效 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明依赖关系、测试难处、模块边界；如「A 直接 new B」「难以 Mock」 |
| **方法** | 要求依赖注入；接口抽象；提供构造函数注入示例 |
| **应用** | 引入依赖注入；生成接口；解耦模块 |
| **提示词范例** | 「NetworkManager 直接 new GameConfig，难以单元测试，请改为依赖注入，并给出接口定义」 |

## 消除重复代码

| 维度 | 内容 |
|------|------|
| **作用** | 消除重复代码，提高复用性 |
| **应用场景** | 重复代码块；多处相似逻辑；复制粘贴 |

| 问题 | 解决方向 |
|------|----------|
| **过度抽象** | 真正需要时抽象；三次原则；保持简单；不过度设计 |

| 要点和思考方向 |
|----------------|
| 适度抽象；用函数、类、模板消除重复 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明重复位置、差异点、抽象风险；如「3 处相似校验」「仅参数不同」「是否过度抽象」 |
| **方法** | 遵循三次原则；要求参数化；避免过度设计；提供抽象建议 |
| **应用** | 提取公共函数；参数化差异；生成模板/泛型 |
| **提示词范例** | 「这段校验逻辑在 Player 和 Monster 中各有一份，差异仅类型，请提取为泛型方法并说明是否值得抽象」 |

## 移动方法

| 维度 | 内容 |
|------|------|
| **作用** | 将方法放到最合适的类中，遵循单一职责 |
| **应用场景** | 方法不属于当前类；被其他类频繁使用；职责不清晰 |

| 问题 | 解决方向 |
|------|----------|
| **职责划分** | 单一职责；方法操作类数据；考虑调用者；DDD |

| 要点和思考方向 |
|----------------|
| 方法放在最合适的类；考虑职责和调用关系 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明方法用途、调用者、数据归属；如「计算伤害在 Player 里但用 Monster 数据」「多处调用」 |
| **方法** | 要求单一职责；方法操作类数据；考虑调用者便利；提供移动影响分析 |
| **应用** | 识别 misplaced 方法；生成移动方案；分析调用影响 |
| **提示词范例** | 「DamageCalculator 的 Calculate 方法在 Player 类中但主要用 Monster 数据，请评估是否应移到 Monster 或独立类」 |

## 单元测试

| 维度 | 内容 |
|------|------|
| **作用** | 测试最小单元（函数、方法），验证行为是否符合预期 |
| **应用场景** | 重构安全网；验证正确性；防止回归；文档化行为 |

**会遇到哪些问题？用什么解决？**

| 问题 | 解决方向 |
|------|----------|
| **测试覆盖不足** | 覆盖率工具；边界与异常；关键路径；定期审查；关注质量 |
| **测试难以编写** | 依赖注入；Mock；降低耦合；测试框架；重构提高可测性 |
| **测试维护成本** | 保持简洁；测试工具；定期重构；测试更简单；数据构建器 |

## 测试模式

## Arrange, Act, Assert (AAA Pattern)

| 维度 | 内容 |
|------|------|
| **作用** | 测试的标准结构模式，提高可读性 |
| **应用场景** | 所有单元测试 |

## 结构

| 阶段 | 说明 |
|------|------|
| Arrange（准备） | 准备测试数据和环境 |
| Act（执行） | 执行被测试的代码 |
| Assert（断言） | 验证结果是否符合预期 |

| 要点和思考方向 |
|----------------|
| AAA 结构清晰；每测试只测一事；测试独立；单元测试是重构安全网 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明被测函数、框架、依赖；如「C# xUnit」「需 Mock 网络」「边界和异常」 |
| **方法** | 要求 AAA 结构；Arrange 准备 Mock；每测试只测一事；提供断言建议 |
| **应用** | 生成单元测试；补充边界用例；设计 Mock 策略 |
| **提示词范例** | 「为 GetPlayerLevel(exp) 写单元测试，用 AAA 结构，覆盖边界（0、临界值、负数）和正常情况，C#」 |

## 测试驱动开发（TDD）

| 维度 | 内容 |
|------|------|
| **作用** | 先写测试再写实现，测试驱动开发 |
| **应用场景** | 需明确需求；需高质量；需快速反馈 |

## 流程

| 阶段 | 说明 |
|------|------|
| Red | 写一个失败的测试 |
| Green | 写最简单的实现让测试通过 |
| Refactor | 重构代码，保持测试通过 |

**会遇到哪些问题？用什么解决？**

| 问题 | 解决方向 |
|------|----------|
| **学习曲线** | 小项目练习；理解价值；逐步采用；团队学习 |
| **时间成本** | 长期节省；减少调试；减少重构风险；提高质量 |

| 要点和思考方向 |
|----------------|
| 帮助明确需求和设计；先写测试提前发现问题；不适用所有场景 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求场景、TDD 适用性；如「新技能系统」「需明确接口」「UI 原型阶段」 |
| **方法** | 要求先写测试用例；Red-Green-Refactor；提供测试用例设计 |
| **应用** | 用 TDD 设计接口；生成首轮测试；指导 Refactor 步骤 |
| **提示词范例** | 「用 TDD 实现技能冷却逻辑，先写测试：冷却中返回剩余时间、冷却完成返回 0、异常输入处理，再写实现」 |
