## 功能系统

| 维度 | 内容 |
|------|------|
| **作用** | 服务端的基础功能系统 |
| **应用场景** | 所有服务端系统、数据库访问、日志记录、配置管理 |
| **做什么的？** | 服务端的基础功能系统 |
| **在哪用？** | 所有服务端系统 |

| 系统 | 问题 | 解决方向 |
|------|------|----------|
| **数据库系统** | 数据库访问性能和管理问题 | 连接池：平衡池子数量（太小阻塞、太大压力大）；异常处理：重试、超时、日志；延迟控制 100ms；ORM：对象关系映射简化操作；持久化：合并、延迟、批量写入；调试：方便查询、修改、分析 |
| **日志系统** | 日志记录、查询和分析问题 | 支持海量数据 IO 不对业务造成压力（异步、缓冲、收集服务）；快速查询（索引、搜索、分析工具）；统计分析（聚合、分析、统计报告）；ELK：Elasticsearch 存储搜索、Logstash 收集、Kibana 查看 |
| **配置系统** | 配置管理和更新问题 | 配表数据：excel 2 json；实现配置中心；支持配置热更新；实现配置版本管理；实现配置校验 |

| ELK 组件 | 功能 |
|----------|------|
| **Elasticsearch** | 日志存储和搜索；分布式搜索、实时查询、数据聚合 |
| **Logstash** | 日志收集；数据转换、数据过滤 |
| **Kibana** | 查看日志；可视化展示、日志查询、数据分析 |

| 要点和思考方向 |
|----------------|
| 数据库系统要平衡性能和资源使用 |
| 日志系统要支持海量数据处理和快速查询 |
| 配置系统要支持热更新和版本管理 |

## Actor模型

| 维度 | 内容 |
|------|------|
| **作用** | 轻量级进程+分布式计算+基于消息的编程模型 |
| **应用场景** | 封闭性高的系统、高并发场景、分布式系统 |
| **做什么的？** | 轻量级进程+分布式计算+基于消息的编程模型 |
| **在哪用？** | 封闭性高的系统 |

| 问题 | 解决方向 |
|------|----------|
| **什么时候使用Actor模型？** 需要判断是否适合使用 Actor 模型 | 适用性：封闭性高的系统（内部状态封闭、通过消息通信、避免共享状态） |
| **如何实现Actor模型？** 需要实现 Actor 模型系统 | 实现 actor system、go routine 池、message handler、context；支持分布式、可伸缩、无锁化、集群化 |

| 应用场景 | 说明 |
|----------|------|
| **玩家数据** | 每个玩家一个 Actor；隔离玩家数据、避免数据竞争 |
| **公会** | 每个公会一个 Actor；管理公会数据、避免并发冲突 |
| **排行榜** | 排行榜 Actor；管理数据、保证一致性 |
| **房间** | 每个房间一个 Actor；管理房间状态、隔离房间数据 |

| 特性 | 说明 |
|------|------|
| **分布式计算** | 支持分布式部署；Actor 路由、跨节点通信、负载均衡 |
| **可伸缩** | 动态创建 Actor、支持 Actor 迁移 |
| **无锁化编程** | 使用消息队列、避免共享状态、消息顺序处理 |
| **集群化管理** | Actor 发现、故障转移、负载均衡 |
| **代码无割裂** | 统一消息处理、清晰 Actor 边界 |

| 实现组件 | 说明 |
|----------|------|
| **actor system** | 管理 Actor 生命周期、消息路由、Actor 调度 |
| **go routine 池** | 管理协程资源、控制并发数量 |
| **message handler** | 处理 Actor 消息、消息分发、消息过滤 |
| **context** | 传递上下文、管理 Actor 状态、超时控制 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明语言、场景；如「Actor」「消息」「玩家」「房间」「无锁」 |
| **方法** | 让 AI 设计 Actor 系统、消息路由、玩家/房间 Actor 划分 |
| **应用** | Actor 架构设计、消息处理、分布式 Actor |
| **提示词范例** | 「游戏服用 Actor 模型，每个玩家一个 Actor，请设计 Go 或 Erlang 下 Actor 生命周期、消息队列、以及玩家与房间 Actor 的通信方式」 |

## 更新

| 维度 | 内容 |
|------|------|
| **作用** | 服务端更新和热更新机制 |
| **应用场景** | 游戏运营和维护阶段、版本更新、功能修复、配置更新 |
| **做什么的？** | 服务端更新和热更新机制 |
| **在哪用？** | 游戏运营和维护阶段 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现不停服更新？** 需要在不中断服务的情况下更新服务器 | 蓝绿更新、灰度更新、滚动更新；实现版本管理、流量切换、回滚机制；监控更新过程 |
| **如何实现代码热更新？** 需要在不重启服务的情况下更新代码 | Lua 脚本（参考 skynet）；使用脚本语言（Lua、Python 等）；实现动态加载；支持代码版本管理；实现安全的热更新流程 |
| **更新管理** 需要管理更新流程和配置 | 白名单：测试号/IP 可进入服务测试；配置管理：配置中心、热更新、版本管理；更新审批、日志记录、回滚机制 |

| 更新策略 | 优点 | 缺点 | 实现 |
|----------|------|------|------|
| **蓝绿更新** | 零停机时间、快速回滚 | 需要双倍资源 | 准备新环境、切换流量、验证后下线旧环境 |
| **灰度更新** | 风险可控、逐步验证 | 更新周期较长 | 部分节点更新、验证、逐步推广 |
| **滚动更新** | 资源占用少、更新平滑 | 更新周期较长 | 逐个节点更新、保持服务可用 |

| 要点和思考方向 |
|----------------|
| 不停服更新是保证服务可用性的重要手段 |
| 选择合适的更新策略很重要 |
| 代码热更新可以提高更新效率 |
| 建立完善的更新管理流程 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明部署环境、策略；如「蓝绿」「灰度」「滚动」「热更新」「不停服」 |
| **方法** | 让 AI 设计更新流程、流量切换、回滚与热更新方案 |
| **应用** | 更新策略设计、K8S 部署、热更新实现 |
| **提示词范例** | 「游戏服需不停服更新，请对比蓝绿、灰度、滚动更新的适用场景，并给出 K8S 下滚动更新的 Deployment 配置和优雅关闭的实现要点」 |
