## 不同游戏类型的考虑

| 维度 | 内容 |
|------|------|
| **作用** | 根据游戏类型选择适合的架构方案 |
| **应用场景** | 游戏项目初期架构设计、技术选型、性能规划 |
| **做什么的？** | 根据游戏类型选择适合的架构方案 |
| **在哪用？** | 游戏项目初期架构设计 |

| 问题 | 解决方向 |
|------|----------|
| **不同类型的游戏对服务端有什么不同的要求？** 不同游戏类型有不同的性能和技术要求 | 基于会话：低延迟、高实时性、状态同步；回合制：容忍较高延迟、保证数据一致性；MMORPG：高并发、高吞吐、数据持久化 |
| **如何根据游戏类型选择合适的架构？** 需要根据游戏类型选择匹配的架构方案 | 分析游戏实时性要求；评估并发和吞吐需求；考虑数据持久化需求；评估延迟容忍度；选择匹配的框架和架构模式 |

| 游戏类型 | 举例 | 要求 | 架构特点 |
|----------|------|------|----------|
| **基于会话——权威服务器** | MOBA、FPS、FTG | 多人对战，高实时，延迟 50~100ms | 低延迟、高实时性、状态同步 |
| **回合制或轻量级实时** | SLG、TCG、TBS | 多人对战，低实时，延迟不超过 300ms | 容忍较高延迟，保证数据一致性 |
| **大型多人在线RPG** | MMORPG、AVG、ACT | 多人同时在线，读写吞吐要求大 | 高并发、高吞吐、数据持久化 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明游戏类型、实时性、并发量；如「MOBA」「MMORPG」「回合制」「延迟要求」 |
| **方法** | 让 AI 根据游戏类型推荐架构模式、技术选型、性能指标 |
| **应用** | 架构设计、技术选型、性能规划 |
| **提示词范例** | 「做 SLG 回合制手游，延迟可接受 300ms，请说明服务端架构应侧重哪些方面，以及与 MOBA 的差异」 |

## 微服务架构

| 维度 | 内容 |
|------|------|
| **作用** | 将服务拆分为多个独立的微服务 |
| **应用场景** | 大型、复杂的游戏项目、需要高可扩展性的项目、多团队协作的项目 |
| **做什么的？** | 将服务拆分为多个独立的微服务 |
| **在哪用？** | 大型、复杂的游戏项目 |

| 问题 | 解决方向 |
|------|----------|
| **是否应该使用微服务架构？** 需要评估是否适合使用微服务架构 | 建议：微服务构建维护代价高，不建议中小规模项目使用；评估项目规模和复杂度、团队规模和能力；考虑运维成本；无服务器架构（Serverless）为另一种选择 |
| **如何实现服务注册和发现？** 微服务需要服务注册和发现机制 | 候选：consul、etcd、zookeeper、eureka；客户端发现/服务端发现；实现服务健康检查；处理服务故障转移 |
| **如何实现负载均衡？** 需要将请求分发到多个服务器 | 使用负载均衡器（Nginx、HAProxy）；实现负载均衡算法（轮询、加权、最少连接等）；实现健康检查；处理服务器故障 |
| **微服务通信** 微服务之间需要通信 | 消息编码：protobuf、json；请求/响应：同步通信；异步消息：异步通信；实现服务间认证和授权 |
| **容器和集群管理** 需要管理大量微服务实例 | 容器：实现快速部署；K8S 管理集群；实现自动扩缩容 |
| **可插拔接口** 需要支持插件化扩展 | 设计插件接口规范；实现插件加载机制；支持插件热更新 |

| 组件 | 说明 |
|------|------|
| **服务注册/发现** | 候选：consul、etcd、zookeeper、eureka；客户端发现/服务端发现 |
| **负载均衡** | 分发请求到多个服务器 |
| **消息编码** | protobuf、json |
| **通信模式** | 请求/响应（同步）、异步消息 |
| **容器** | 实现快速部署 |
| **K8S** | Master：API Server、Scheduler、Controller manager、etcd；Node：kubelet、kube-proxy、Docker Engine |
| **可插拔接口** | 支持插件化扩展 |
| **候选框架** | Go Micro |

| 要点和思考方向 |
|----------------|
| 微服务框架看似清晰、明确、容易伸缩，但构建维护代价高昂，不建议中小规模项目使用 |
| 无服务器架构（Serverless）为另一种架构选择 |
