## 命令工具

| 维度 | 内容 |
|------|------|
| **作用** | 提供通过命令行调用的功能模块，供开发者直接使用，或作为构建系统、CI、脚本的对接接口 |
| **应用场景** | 开发者直接使用、构建系统集成、CI/CD 流程集成、脚本自动化 |

| 问题 | 解决方向 |
|------|----------|
| **如何设计命令工具** 需要设计清晰、稳定的命令接口 | 命令参数要清晰、稳定，适合被脚本和自动化工具调用；可读的帮助信息（--help）、错误码设计；设计命令接口规范；实现参数解析、错误处理 |
| **如何使用命令工具** 需要支持不同的使用方式 | 低频：手动调用（交互式、详细输出、调试模式）；高频：集成到构建工具或 CI 流程（非交互、静默、返回码） |
| **如何实现命令工具功能** 需要实现各种命令工具功能 | 数据格式转换（Excel→JSON、JSON→二进制）；资源检查（命名规范、引用完整性、尺寸/格式检查） |

| 举例 | 说明 |
|------|------|
| **数据格式转换脚本** | Excel→JSON、JSON→二进制配置 |
| **资源检查工具** | 命名规范、引用完整性、尺寸/格式检查等 |

| 要点和思考方向 |
|----------------|
| 命令工具是工具链的基础组件 |
| 设计清晰的接口便于集成和使用 |
| 支持多种使用方式提高灵活性 |
| 优化工具性能保证执行效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「命令行」「参数」「--help」「错误码」「Excel→JSON」「资源检查」 |
| **方法** | 让 AI 设计 CLI 接口、参数解析、错误处理、脚本化调用 |
| **应用** | 命令工具设计、数据转换脚本、资源检查工具 |
| **提示词范例** | 「需开发命令行工具：Excel 转 JSON、JSON 转二进制配置，请设计清晰的参数接口、--help 输出、错误码约定，以及适合 CI 调用的非交互模式」 |

## 构建工具

| 维度 | 内容 |
|------|------|
| **作用** | 通过对一系列工具/插件的编排调用，自动化完成一批构建和打包工作 |
| **应用场景** | 项目构建、打包发布、自动化流程、CI/CD 集成 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现构建自动化** 需要自动化完成构建和打包工作 | 通常由外界触发（命令行、CI、任务调度系统）；常被集成到 CI/CD 流水线中；设计构建流程；实现构建脚本 |
| **如何管理依赖** 需要管理复杂的依赖关系 | 管理不同模块/库之间的依赖关系；确保构建顺序正确；实现依赖排序、并行构建 |
| **如何管理版本** 需要对构建产物进行版本管理 | 对构建产物进行版本标记；与代码仓库版本联动（Tag、Branch 等） |
| **如何选择构建工具** 需要选择合适的构建工具 | 评估工具特性、项目需求；选择合适工具 |

| 常见工具 | 特点 |
|----------|------|
| **ant** | Java 构建工具；XML 配置；跨平台支持 |
| **maven** | 项目管理和构建工具；依赖管理；插件系统 |
| **gradle** | 灵活的构建工具；Groovy/Kotlin DSL；增量构建 |
| **make** | 传统构建工具；Makefile 配置；广泛支持 |

| 要点和思考方向 |
|----------------|
| 构建工具是自动化流程的核心 |
| 实现完善的依赖管理保证构建正确性 |
| 实现版本管理便于追踪和管理 |
| 选择合适的构建工具提高开发效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Gradle」「Make」「依赖」「并行构建」「版本标记」「CI 集成」 |
| **方法** | 让 AI 设计构建流程、依赖排序、版本联动、脚本编排 |
| **应用** | 构建脚本、依赖管理、版本管理 |
| **提示词范例** | 「游戏项目需构建脚本，支持多模块、依赖顺序、并行构建，请用 Gradle 或 Make 设计构建流程，并实现构建产物版本标记与 Git Tag 联动」 |

## 持续集成工具

| 维度 | 内容 |
|------|------|
| **作用** | 运行长期在线服务，在指定触发条件下（代码提交、定时任务），自动完成构建、测试、部署等工作 |
| **应用场景** | 自动化构建、自动化测试、自动化部署、持续集成流程 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现持续集成** 需要实现自动化构建、测试、部署流程 | 实现触发机制、构建流程、测试流程、部署流程 |
| **如何选择 CI 工具** 需要选择合适的 CI 工具 | Jenkins（开源、丰富插件、广泛使用）；Hudson（Jenkins 前身） |
| **如何实现流水线** 需要将工具和脚本串联成流水线 | 将「命令工具 + 构建脚本」串联形成固定流水线；设计流水线流程；实现工具串联 |
| **如何实现结果展示和通知** 需要对构建和测试结果进行展示和通知 | 可视化展示与通知（邮件、IM、看板） |

| 工具 | 特点 |
|------|------|
| **Jenkins** | 开源 CI 工具；丰富的插件；广泛使用 |
| **Hudson** | 开源 CI 工具；Jenkins 的前身 |

| 要点和思考方向 |
|----------------|
| 持续集成工具是自动化流程的核心 |
| 实现完善的流水线提高开发效率 |
| 实现结果展示和通知便于问题追踪 |
| 选择合适的 CI 工具满足项目需求 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Jenkins」「流水线」「触发」「构建」「测试」「通知」「邮件」 |
| **方法** | 让 AI 设计 CI 流水线、触发条件、工具串联、通知配置 |
| **应用** | CI 流水线、构建触发、结果通知 |
| **提示词范例** | 「需 Jenkins 流水线：代码提交触发、构建→测试→部署，请设计 Pipeline 脚本，串联命令工具和构建脚本，并配置构建失败时邮件/IM 通知」 |

## 开发者工具

| 维度 | 内容 |
|------|------|
| **作用** | 为开发者提供一套丰富的命令行环境和工具集合 |
| **应用场景** | 开发环境、脚本编写、自动化流程、跨平台开发 |

| 问题 | 解决方向 |
|------|----------|
| **如何提供工具集合** 需要提供丰富的工具集合 | cygwin：包含大量 Linux 工具命令；为产品生产线提供丰富的工具集合，方便编写跨平台脚本与自动化流程 |
| **如何统一工具环境** 需要统一团队的工具环境 | 尽量统一团队使用的基础工具集合（Shell、Python 版本、命令工具等）；建立工具规范 |
| **如何降低使用门槛** 需要降低工具使用门槛 | 通过脚本封装复杂命令；实现脚本库；提供使用文档 |

| 工具 | 说明 |
|------|------|
| **cygwin** | 包含大量 Linux 工具命令；跨平台支持；方便编写跨平台脚本与自动化流程 |

| 要点和思考方向 |
|----------------|
| 开发者工具是开发效率的基础 |
| 统一工具环境提高协作效率 |
| 通过脚本封装降低使用门槛 |
| 选择合适的工具集满足开发需求 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「cygwin」「Shell」「Python」「跨平台脚本」「工具规范」 |
| **方法** | 让 AI 设计脚本封装、环境统一、跨平台兼容方案 |
| **应用** | 开发环境、脚本库、工具规范 |
| **提示词范例** | 「团队需统一开发环境：Shell、Python 版本、常用命令，请设计跨平台脚本封装方案（Windows/Linux/macOS），以及团队工具清单与脚本库结构」 |

## 测试自动化

| 维度 | 内容 |
|------|------|
| **作用** | 用自动化方式替代大量重复的手工测试工作，提升效率和质量 |
| **应用场景** | 功能测试、回归测试、性能测试、兼容性测试 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现测试自动化** 需要实现自动化测试 | 实现测试框架、测试脚本、测试执行、测试报告 |
| **如何提高测试效率** 需要提高测试效率 | 自动执行测试用例减少手动工作量；在较短时间内执行大量测试案例；实现并行测试 |
| **如何确保测试质量** 需要确保测试质量 | 确保各功能和模块在不同平台和环境下的稳定性和一致性；重复执行相同测试保证版本间行为一致 |
| **如何提高测试可靠性** 需要提高测试可靠性 | 减少人为错误；精确模拟用户操作和各种场景 |
| **如何加速发布周期** 需要加速发布周期 | 每次代码更改后自动执行测试快速反馈；缩短测试周期加速版本发布 |
| **如何降低长期成本** 需要降低长期成本 | 初期投入脚本开发，长期减少人力成本 |
| **如何选择测试工具** 需要选择合适的测试工具 | 评估工具特性、项目需求；选择合适工具 |

| 带来的价值 | 说明 |
|------------|------|
| **提高效率** | 自动化执行、批量测试 |
| **确保质量** | 稳定性测试、回归测试 |
| **提高可靠性** | 减少人为错误、场景模拟 |
| **加速发布周期** | 快速反馈、周期缩短 |
| **降低长期成本** | 长期减少人力成本 |

| 工具 | 说明 |
|------|------|
| **appium** | 开源自动化工具；使用 WebDriver 在 Android 或 iOS 上运行脚本和测试；支持原生应用、移动 Web 应用和混合应用 |

| 要点和思考方向 |
|----------------|
| 测试自动化是质量保证的重要手段 |
| 提高测试效率和质量 |
| 选择合适的测试工具满足项目需求 |
| 平衡初期投入和长期收益 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Appium」「自动化测试」「回归」「并行」「测试报告」 |
| **方法** | 让 AI 设计测试框架、用例组织、自动化脚本、报告生成 |
| **应用** | 自动化测试、回归测试、性能测试 |
| **提示词范例** | 「游戏需自动化测试：功能回归、多平台，请对比 Appium 与 Unity Test Framework，设计测试用例组织方式、并行执行与测试报告生成流程」 |

## 其它自动化工具

| 维度 | 内容 |
|------|------|
| **作用** | 打通各种第三方 Web 应用或服务，串联形成自动流程 |
| **应用场景** | 工作流自动化、服务集成、通知自动化、数据自动化 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现服务集成** 需要集成各种第三方服务 | 实现服务接口、服务连接、数据转换 |
| **如何实现自动化流程** 需要实现自动化工作流 | 内容或数据的自动化创作、管理与发布；生成报表后自动发送邮件/推送到 IM 群；构建完成后自动通知相关负责人并附带链接 |
| **如何选择自动化工具** 需要选择合适的自动化工具 | 评估工具特性、服务支持；选择合适工具 |

| 应用举例 | 说明 |
|----------|------|
| **通知自动化** | 生成报表后自动发送邮件/推送到 IM 群 |

|----------------|

| 实现服务集成形成自动化流程 |
| 选择合适的工具满足业务需求 |
| 优化自动化流程提高效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Zapier」「Power Automate」「通知」「构建完成」「邮件」「IM」 |
| **方法** | 让 AI 设计工作流自动化、服务集成、通知触发 |
| **应用** | 通知自动化、构建通知、报表推送 |
| **提示词范例** | 「构建完成后需自动通知：邮件/IM 附带构建链接、失败原因，请评估 Zapier 与 Jenkins 插件方案，设计通知触发与推送流程」 |
