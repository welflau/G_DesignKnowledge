## 图谱

## 系统概览

| 维度 | 内容 |
|------|------|
| **作用** | 提供完善的UI管理系统，支持游戏所需的各种UI特性 |
| **应用场景** | 所有需要UI界面的游戏 |
| **核心功能** | UI分层和分类管理；页面切换和流程管理；UI资源加载和释放；事件处理和交互响应；动画和特效支持 |

| 问题 | 解决方向 |
|------|----------|
| **UI分层管理混乱** UI元素层级关系不清晰，导致显示错乱、事件响应异常 | 建立清晰的UI层级体系（背景层、内容层、弹窗层、提示层）；使用层级管理器统一管理；明确不同层级UI的交互规则 |
| **页面切换不流畅** 页面切换时出现卡顿、闪烁、资源加载延迟 | 实现页面切换动画；预加载常用页面资源；使用异步加载；合理管理页面生命周期 |
| **页面流程不清晰** 页面之间的跳转关系混乱，用户容易迷失 | 采用树状结构的UI页面关系；使用堆栈化管理页面进出；提供清晰的导航路径；记录页面跳转历史，支持返回 |
| **资源管理不当** UI资源加载时机不当，导致内存占用过高或加载延迟 | 规划好各个UI的加载时机（预加载 vs 动态加载）；及时释放不用的UI资源；使用对象池复用UI元素；建立资源依赖关系管理 |

| 要点和思考方向 |
|----------------|
| 要对UI做好分层、分类管理，建立清晰的层级体系 |
| 要支持页面切换的动画表现，提升用户体验 |
| 能让页面有清晰的流程和结构，采用树状结构和堆栈化管理 |
| 合理规划资源加载时机，平衡内存占用和加载速度 |
| 使用MVC等设计模式，保持代码结构清晰 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「分层」「页面切换」「流程」「资源加载」「MVC」 |
| **方法** | 让 AI 设计 UI 架构、层级体系；或生成页面管理器、堆栈管理、MVC 结构 |
| **应用** | UI 架构设计、页面流程、资源规划 |
| **提示词范例** | 「游戏 UI 需要分层管理（背景、内容、弹窗、提示），请设计 C# 层级管理器和页面切换的堆栈结构」 |

## UI类型分类

### Page（页面）

| 维度 | 内容 |
|------|------|
| **作用** | 整个场景的UI容器，通常占据整个屏幕 |
| **应用场景** | 主菜单页面、游戏大厅页面、设置页面、商店页面等主要功能页面 |

| 问题 | 解决方向 |
|------|----------|
| **页面切换管理** 多个页面之间的切换关系复杂，容易出现状态混乱 | 使用页面管理器统一管理页面切换，支持堆栈化进出 |

| 要点和思考方向 |
|----------------|
| Page通常作为场景的主要UI容器，需要清晰的进入和退出逻辑 |
| 支持页面切换动画，提升用户体验 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「页面切换」「堆栈」「进入退出」「动画」 |
| **方法** | 让 AI 设计页面管理器、生成页面基类；或实现堆栈化进出、切换动画 |
| **应用** | 主菜单、大厅、设置、商店等主页面 |
| **提示词范例** | 「Page 页面需要堆栈化管理，支持 Push/Pop 和切换动画，请给出 Unity C# 页面管理器实现和 Page 基类接口」 |

### Window（窗口）

| 维度 | 内容 |
|------|------|
| **作用** | 场景中会出现的窗口，通常有背景遮罩，可以关闭 |
| **应用场景** | 功能窗口（背包、角色、装备等）、设置窗口、信息展示窗口 |

| 问题 | 解决方向 |
|------|----------|
| **窗口层级管理** 多个窗口同时打开时，层级关系混乱 | 使用窗口管理器管理窗口层级，支持置顶、置底等操作 |
| **窗口关闭逻辑** 窗口关闭时数据未保存、状态未清理 | 实现窗口生命周期管理，在关闭时自动清理资源 |

| 要点和思考方向 |
|----------------|
| Window通常有明确的打开和关闭操作，需要管理好生命周期 |
| 支持窗口拖拽、缩放等交互（根据需求） |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「窗口层级」「生命周期」「拖拽」「遮罩」 |
| **方法** | 让 AI 设计窗口管理器、生成 Window 基类；或实现置顶、关闭清理、拖拽逻辑 |
| **应用** | 背包、角色、装备、设置等窗口 |
| **提示词范例** | 「Window 需要层级管理、打开关闭生命周期，请设计 C# 窗口管理器，支持置顶、关闭时自动释放资源」 |

### Dialog（对话框）

| 维度 | 内容 |
|------|------|
| **作用** | 可弹出，有特定确认取消行为的UI，用于提示、确认等操作 |
| **应用场景** | 确认对话框、提示对话框、输入对话框 |

| 问题 | 解决方向 |
|------|----------|
| **模态化处理** 对话框与其他UI的关系不明确 | 设计清晰的对话框管理机制；明确同时展示几个对话框的规则；使用遮罩层阻止底层UI交互 |
| **对话框队列** 多个对话框需要依次显示，但显示逻辑混乱 | 使用对话框队列，按顺序显示对话框 |

| 要点和思考方向 |
|----------------|
| Dialog需要明确的确认/取消行为定义 |
| 支持对话框队列，避免同时显示多个对话框 |
| 考虑对话框的优先级和显示规则 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「确认取消」「模态」「队列」「遮罩」 |
| **方法** | 让 AI 设计对话框管理器、生成 Dialog 基类；或实现队列、优先级、回调接口 |
| **应用** | 确认框、提示框、输入框 |
| **提示词范例** | 「Dialog 需要队列管理，按顺序显示，支持确认/取消回调，请给出 C# 对话框队列实现和模态遮罩逻辑」 |

| 要点和思考方向 |
|----------------|
| Loading界面需要提供清晰的进度反馈 |
| 支持取消加载操作（根据场景） |
| 优化加载体验，避免用户等待焦虑 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「进度」「取消」「平滑」「提示信息」 |

### Info（信息提示）

| 维度 | 内容 |

| 要点和思考方向 |
|----------------|
| Info通常自动消失，需要合理设置显示时间 |
| 支持信息队列，避免信息堆积 |
| 考虑信息的优先级和重要性 |

| 类型 | AI Coding 指南 |

## 进场与离场

| 维度 | 内容 |
|------|------|
| **作用** | UI显示和隐藏时的动画表现，提升用户体验 |
| **应用场景** | 所有UI页面的显示和隐藏 |

| 问题 | 解决方向 |
|------|----------|
| **动画性能问题** 复杂的进场离场动画可能导致性能下降 | 使用高效的动画库（如DOTween、iTween）；优化动画复杂度；在低端设备上简化或关闭动画 |
| **动画时机控制** 动画与数据加载时机不匹配 | 在数据加载完成后再播放进场动画；使用异步加载；实现动画队列按顺序播放 |

| 动画表现内容 | 说明 |
|--------------|------|
| **全屏过场效果** | 页面切换时的全屏过渡动画 |
| **页面的动态效果** | 整个页面的进入/退出动画 |
| **页面上各个元素的动态效果** | 页面内元素的逐个显示动画 |

| 要点和思考方向 |
|----------------|
| 进场和离场要有动态表现，提升用户体验 |
| 动画要流畅自然，符合游戏风格 |
| 考虑性能影响，在低端设备上适当简化 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「进场」「离场」「DOTween」「全屏过场」 |
| **方法** | 让 AI 生成进场离场动画序列；或设计动画时机、低端设备降级、DOTween 配置 |
| **应用** | 页面切换、窗口打开关闭、元素逐个显示 |
| **提示词范例** | 「UI 进场需要全屏过场 + 页面元素逐个动画，请给出 Unity DOTween 序列实现和与数据加载时机的配合方式」 |

## 加载与释放

| 维度 | 内容 |
|------|------|
| **作用** | 管理UI资源的加载和释放时机，优化内存使用 |
| **应用场景** | 所有UI页面的资源管理 |

| 问题 | 解决方向 |
|------|----------|
| **加载时机不当** 导致显示延迟或内存占用过高 | 规划好各个UI的加载时机；使用预加载和动态加载相结合；实现异步加载 |
| **资源释放不及时** UI关闭后资源未释放，导致内存泄漏 | 实现自动释放机制；在UI关闭时自动释放资源；使用引用计数管理资源生命周期 |
| **资源重复加载** 相同资源被多次加载 | 实现资源缓存机制；使用资源管理器统一管理；建立资源引用计数 |

| 加载策略 | 说明 |
|----------|------|
| **预加载** | 预定义加载时机（游戏启动、进入场景）；动态判断预先加载的时机 |
| **动态加载** | 使用时进行加载和创建，适合不常用的UI |

| 要点和思考方向 |
|----------------|
| 需要管理UI资源的加载和释放时机，平衡内存占用和加载速度 |
| 实现资源缓存和复用机制，避免重复加载 |
| 使用异步加载，避免阻塞主线程 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「预加载」「动态加载」「缓存」「引用计数」 |
| **方法** | 让 AI 设计 UI 资源管理器、生成加载策略；或实现异步加载、自动释放、缓存复用 |
| **应用** | UI 资源生命周期、内存优化 |
| **提示词范例** | 「UI 资源需要预加载常用、动态加载不常用，关闭时释放，请设计 C# 资源管理器和引用计数释放逻辑」 |

## 资源管理

| 维度 | 内容 |
|------|------|
| **作用** | 管理UI资源的加载时机和方式，优化内存使用和加载性能 |
| **应用场景** | 所有UI系统都需要资源管理 |

| 问题 | 解决方向 |
|------|----------|
| **加载时机规划困难** 如何平衡内存占用和加载速度 | 分析UI使用频率，高频预加载、低频动态加载；在合适时机预加载；建立UI加载优先级机制 |
| **预加载 vs 动态加载选择** | 预加载：高频UI、启动必需UI、可预判使用的UI；动态加载：低频UI、体积较大、不确定会使用的UI |
| **资源依赖管理** 依赖关系复杂 | 建立资源依赖关系图；按依赖顺序加载；实现依赖资源自动加载 |
| **图集管理** UI图片资源分散，DrawCall过多 | 使用图集将多个小图片打包；按功能模块组织图集；动态图集和静态图集相结合 |

| 要点和思考方向 |
|----------------|
| 规划好各个UI的加载时机，平衡内存占用和加载速度 |
| 预加载常用UI，动态加载不常用UI |
| 建立资源缓存机制，使用图集优化渲染性能 |
| 实现资源依赖管理，确保资源加载顺序正确 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「加载时机」「图集」「依赖」「DrawCall」 |
| **方法** | 让 AI 设计加载优先级、图集划分；或生成依赖图、预加载时机判断 |
| **应用** | 资源规划、图集管理、DrawCall 优化 |
| **提示词范例** | 「UI 图集如何按功能模块划分？请说明 UGUI 图集与 DrawCall 的关系，以及预加载与动态加载的时机规划」 |

| **提示词范例** | 「页面栈需要支持 Push/Pop/清空/跳转，返回时恢复滚动位置，请给出 C# 栈管理实现和状态保存方案」 |

## 层级管理

| 维度 | 内容 |
|------|------|
| **作用** | 管理UI显示层级的关系，确保UI正确显示和交互 |
| **应用场景** | 所有UI系统都需要层级管理 |

| 问题 | 解决方向 |
|------|----------|
| **层级冲突** 多个UI同时显示时层级关系混乱 | 建立清晰的层级体系；使用层级管理器统一管理；明确不同层级的显示规则 |
| **事件响应冲突** 上层UI和下层UI同时响应事件 | 实现事件穿透机制；上层UI阻止下层UI事件响应；使用事件优先级管理 |

## UI系统设计模式

| 维度 | 内容 |
|------|------|
| **作用** | 使用设计模式组织UI代码，提高代码可维护性和可扩展性 |
| **应用场景** | 所有UI系统开发 |

| 模式 | 说明 | 应用场景 |
|------|------|----------|
| **MVC** | Model-View-Controller，分离数据、视图和逻辑 | 复杂的UI功能开发 |
| **MVP** | Model-View-Presenter，View和Model完全解耦 | 需要View和Model完全解耦的场景 |
| **MVVM** | Model-View-ViewModel，通过数据绑定实现View和Model同步 | 需要数据驱动的UI开发 |

| 要点和思考方向 |
|----------------|
| MVC：Model管理数据，View管理显示，Controller管理逻辑；职责明确，避免耦合 |
| MVP：比MVC更解耦，Presenter负责View和Model之间的交互 |
| MVVM：通过数据绑定减少手动更新UI的代码；需要框架支持数据绑定 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明场景、目标；如「MVC」「MVP」「MVVM」「数据绑定」 |
| **方法** | 让 AI 设计 MVC/MVP/MVVM 结构、生成示例；或对比三种模式、说明职责划分 |
| **应用** | UI 代码组织、数据与视图解耦 |
| **提示词范例** | 「Unity UI 使用 MVC，请给出 C# 示例：Model 数据类、View 绑定 UI 组件、Controller 处理逻辑和三者交互方式」 |

|----------------|

| **提示词范例** | 「UGUI 性能优化：图集、Canvas 分组、对象池、批量更新，请给出具体实现建议和 Profiler 排查要点」 |
