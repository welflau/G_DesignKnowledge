<h2 align="center">引擎的基本功能系统</h2>
<p>
游戏引擎为游戏内容制作和开发提供了一整套基础能力，本节聚焦那些「几乎所有项目都会用到」的核心系统：UI、GamePlay、特效、场景与渲染。重点是它们各自解决什么问题、在哪些地方会踩坑，以及在生产流程中的角色，而不是具体 API 细节。
</p>

## 总体作用

| 维度 | 内容 |
|------|------|
| **作用** | 提供与「表现」和「互动」相关的基础功能，让美术和程序站在一个统一的技术地基上协作 |
| **应用场景** | 游戏开发基础框架、美术和程序协作、工具链和自动化流程 |

| 问题 | 解决方向 |
|------|----------|
| **如何复用成熟能力** 需要避免每个项目都从零搭建底层 | 使用成熟的游戏引擎；复用引擎提供的功能系统；建立项目通用框架；选择合适的游戏引擎；建立项目模板 |
| **如何统一系统接口** 需要统一的系统接口便于工具链和自动化流程 | 通过统一的系统接口（UI、事件、渲染、场景等）让工具链和自动化流程更容易搭建；定义统一的系统接口；实现标准化的数据格式 |

| 要点和思考方向 |
|----------------|
| 引擎基本功能系统是游戏开发的基础 |
| 复用成熟能力提高开发效率 |
| 统一系统接口便于工具链和自动化 |
| 建立清晰的协作规范 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、项目；如「Unity」「Unreal」「UI」「事件」「渲染」「接口」 |
| **方法** | 让 AI 设计引擎功能选型、系统接口、统一规范 |
| **应用** | 引擎选型、框架设计、接口规范 |
| **提示词范例** | 「Unity 与 Unreal 在 UI、GamePlay、特效、场景系统上的差异？请对比各系统成熟度、学习成本，以及中小团队 2D/3D 项目的选型建议」 |

## UI 系统

| 维度 | 内容 |
|------|------|
| **作用** | 负责游戏中的信息展示与交互，是玩家与游戏交互的第一入口 |
| **应用场景** | 游戏界面制作、玩家交互、信息展示 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现不同平台的 UI** 不同平台需要不同的 UI 形式 | GUI（窗口、按钮、列表、提示等）；Menu（主菜单、模式选择、设置页）；HUD（血条、弹药数、小地图、状态图标等）；实现平台适配、输入映射 |
| **如何解决 UI 渲染问题** UI 渲染需要处理层级和图集管理 | 层级管理（避免遮挡错乱）；图集管理（减少 DrawCall，统一打包）；实现批量渲染 |
| **如何管理 UI 资源** 需要管理 UI 资源的加载和生命周期 | 资源加载、卸载、缓存策略；使用资源管理系统；实现资源池 |
| **如何管理 UI 对象** 需要管理 UI 节点的创建和销毁 | 对象池；节点复用；减少 GC 压力 |
| **如何实现 UI 交互** 需要实现 UI 的交互功能 | 事件系统（点击、拖拽、悬停、键盘/手柄输入映射） |
| **如何实现 UI 组件化** 需要实现可复用的 UI 组件 | 可复用组件（按钮、进度条、列表单元格等）；组件继承、组件组合 |
| **如何实现 UI 编辑能力** 需要提供 UI 编辑工具 | 可视化 UI 编辑器（所见即所得）；布局系统（锚点、对齐、自动适配不同分辨率） |

| 典型形式 | 适用 | 说明 |
|----------|------|------|
| **GUI** | 电脑、手机等交互能力丰富的平台 | 窗口、按钮、列表、提示等常规组件 |
| **Menu** | 主机游戏等手柄操控的平台 | 主菜单、模式选择、设置页 |
| **HUD** | 信息展示，无需 Input 交互 | 血条、弹药数、小地图、状态图标等 |

| 要点和思考方向 |
|----------------|
| UI 系统是玩家交互的第一入口 |
| 需要支持不同平台和输入方式 |
| 优化 UI 渲染和资源管理性能 |
| 实现组件化提高开发效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「UGUI」「层级」「图集」「对象池」「事件」 |
| **方法** | 让 AI 设计 UI 架构、层级管理、图集策略、对象池 |
| **应用** | UI 架构设计、性能优化、组件化 |
| **提示词范例** | 「Unity UGUI 需层级管理、图集减少 DrawCall，请设计 Canvas 分层、图集划分策略，以及 UI 对象池的创建与回收 C# 实现」 |

## GamePlay 功能

| 维度 | 内容 |
|------|------|
| **作用** | 负责游戏整体的控制与管理，是「游戏逻辑层」的核心能力 |
| **应用场景** | 游戏逻辑实现、游戏流程管理、游戏状态管理 |

| 问题 | 解决方向 |
|------|----------|
| **如何管理游戏流程和状态** 需要管理复杂的游戏流程和状态 | 状态流转（标题→登陆→大厅→战斗→结算→返回）；支持暂停、恢复、重开等；使用状态机模式 |
| **如何实现脚本系统** 需要支持脚本驱动的游戏逻辑 | 使用 Lua、C# 脚本等驱动游戏逻辑；支持热更或快速迭代 |
| **如何实现世界加载** 需要高效加载场景和地图 | 场景/地图的加载与卸载；异步加载、分块加载、Streaming World 等 |
| **如何实现事件/消息系统** 需要实现模块间解耦 | 模块间解耦，通过消息或事件通知；例如角色死亡事件通知 UI、音效、任务系统 |
| **如何管理世界内容** 需要管理静态和动态世界内容 | 静态：地图、机关、出生点、刷怪点等；动态：怪物、道具、可破坏物、NPC 等；使用对象池 |

| 要点和思考方向 |
|----------------|
| GamePlay 功能是游戏逻辑层的核心 |
| 需要支持灵活的脚本系统和热更新 |
| 优化世界加载和对象管理性能 |
| 实现模块解耦提高可维护性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「状态机」「Lua」「事件系统」「异步加载」「对象池」 |
| **方法** | 让 AI 设计游戏流程、事件系统、世界加载、对象管理 |
| **应用** | 游戏流程设计、事件解耦、加载优化 |
| **提示词范例** | 「游戏流程：标题→登录→大厅→战斗→结算，请设计 C# 状态机实现，以及角色死亡时通知 UI、音效、任务系统的事件发布/订阅模式」 |

## 特效系统

| 维度 | 内容 |
|------|------|
| **作用** | 负责各种视觉效果的表现，是「爽感」「反馈感」的重要来源 |
| **应用场景** | 视觉效果制作、游戏氛围营造、玩家反馈 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现各种视觉效果** 需要实现各种视觉效果 | Light Mapping & 阴影；HDR；各类 Post Effects（Bloom、色调映射、景深、运动模糊、色偏）；使用后处理管线；实现特效 LOD |
| **如何实现特效编辑** 需要提供特效编辑工具 | 特效编辑器（粒子系统、轨迹、曲线控制）；与时间轴/动画系统联动（技能释放时的特效时序） |

| 典型效果 | 说明 |
|----------|------|
| **Light Mapping & 阴影** | 光照贴图、阴影系统 |
| **HDR** | 高动态范围、色调映射 |
| **Post Effects** | Bloom、色调映射、景深、运动模糊、色偏 |

| 要点和思考方向 |
|----------------|
| 特效系统是游戏体验的重要来源 |
| 需要平衡视觉效果和性能 |
| 实现灵活的特效编辑工具 |
| 优化特效性能避免影响游戏流畅度 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「粒子」「后处理」「Bloom」「LOD」「特效编辑器」 |
| **方法** | 让 AI 设计特效管线、后处理堆栈、特效 LOD |
| **应用** | 特效配置、性能优化、编辑器扩展 |
| **提示词范例** | 「Unity URP 需 Bloom、色调映射、技能释放特效时序，请说明后处理堆栈配置、粒子系统与 Timeline 的联动，以及特效 LOD 的切换距离建议」 |

## 场景系统

| 维度 | 内容 |
|------|------|
| **作用** | 管理场景中的对象与空间结构，是「世界」这一概念在引擎中的承载者 |
| **应用场景** | 场景制作、对象管理、可见性管理 |

| 问题 | 解决方向 |
|------|----------|
| **如何管理场景对象** 需要管理场景中的大量对象 | 管理场景中的对象（放置、启用/禁用、分组）；使用场景图；实现对象索引 |
| **如何实现可见性管理** 需要管理对象的可见性以提高性能 | 视锥剔除、遮挡剔除、距离剔除；Occlusion Culling、PVS；LOD 系统；BSP Tree 等空间划分结构 |
| **如何实现场景编辑** 需要提供场景编辑工具 | 场景编辑器（摆放物体、设置区域、配置触发器）；分层组织（大场景管理与协作） |

| 典型技术 | 说明 |
|----------|------|
| **Occlusion & PVS** | 遮挡剔除；PVS 预计算可见集 |
| **LOD 系统** | 不同距离使用不同精度模型/贴图 |
| **BSP Tree 等** | 空间划分，用于快速查询、碰撞检测和渲染裁剪 |

| 要点和思考方向 |
|----------------|
| 场景系统是游戏世界的承载者 |
| 需要优化可见性管理提高性能 |
| 实现灵活的场景编辑工具 |
| 支持大场景的管理和协作 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「视锥剔除」「遮挡剔除」「LOD」「PVS」「BSP」 |
| **方法** | 让 AI 设计可见性方案、LOD 配置、场景分块 |
| **应用** | 可见性优化、大场景管理、性能调优 |
| **提示词范例** | 「Unity 大场景需优化可见性，请对比视锥剔除、遮挡剔除、LOD 的适用场景，以及 Occlusion Culling 与 PVS 的配置要点和性能影响」 |

## 渲染系统

| 维度 | 内容 |
|------|------|
| **作用** | 负责所有可见事物的最终呈现，是基础中的基础 |
| **应用场景** | 游戏画面渲染、视觉效果呈现、性能优化 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现材质和 Shader 系统** 需要实现灵活的材质和 Shader 系统 | 材质系统（PBR/非 PBR）；Shader 管线与可扩展机制（自定义 Shader、Shader Graph 等） |
| **如何实现光照系统** 需要实现真实的光照效果 | 静态光照（光照贴图、预计算 GI）；动态光照（动态阴影、实时反射等） |
| **如何实现摄像机系统** 需要支持多种摄像机需求 | 多摄像机支持（主摄像机、UI 摄像机、回放摄像机等）；视口设置（分屏、画中画） |
| **如何管理贴图资源** 需要管理大量贴图资源 | 纹理导入、压缩、Mipmap、图集管理等 |
| **如何封装图形驱动接口** 需要封装底层图形 API | 封装 DirectX/OpenGL/Vulkan/Metal 等 |
| **如何与生产流程结合** 需要与生产流程结合 | 决定美术资源的技术规格；与性能优化高度相关（DrawCall、填充率、带宽等） |

| 要点和思考方向 |
|----------------|
| 渲染系统是游戏画面的基础 |
| 需要平衡视觉效果和性能 |
| 建立清晰的资源规格和规范 |
| 优化渲染性能保证游戏流畅度 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「PBR」「Shader」「光照」「材质」「DrawCall」 |
| **方法** | 让 AI 设计材质规范、Shader 管线、光照方案 |
| **应用** | 渲染管线、材质规范、性能优化 |
| **提示词范例** | 「Unity URP 需 PBR 材质、光照贴图、动态阴影，请说明材质参数规范、光照贴图烘焙流程，以及 DrawCall 与批处理的优化建议」 |
