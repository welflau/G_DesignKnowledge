## 正文
<h2 align="center">引擎的其它系统</h2>
<p>
除了渲染、场景、UI 等「看得见」的部分，引擎内部还有一批支撑性系统，负责资源管理、核心通用能力、跨平台封装以及与第三方 SDK 的衔接。这些系统决定了引擎和产品能否高效开发、稳定运行和长期演进。
</p>

*资源管理,核心系统,跨平台接口,第三方SDK,游戏特有系统,AI Coding*

## 资源管理

| 维度 | 内容 |
|------|------|
| **作用** | 管理所有游戏数字内容在磁盘和运行时中的组织、加载与生命周期 |
| **应用场景** | 游戏资源加载、资源生命周期管理、资源打包和分发、资源依赖管理 |

| 核心认识 |
|----------|
| 所有游戏的数字内容，都要以数据（文件）的形式进行存储和使用 |
| 资源管理决定了项目的可维护性、加载性能和包体结构 |

| 问题 | 解决方向 |
|------|----------|
| **如何管理各种资源类型** 需要管理多种类型的游戏资源 | 3D 模型、贴图、材质、字体、骨骼、碰撞数据、地图/关卡数据；建立资源类型系统；实现统一资源接口 |
| **如何建立资源规范** 需要统一的资源命名和目录规范 | 统一的资源命名与目录规范；建立命名规范、目录结构；实现自动化工具 |
| **如何管理资源依赖关系** 资源之间存在复杂的依赖关系 | 资源依赖关系管理（模型引用的材质、贴图、动画等）；实现依赖分析、依赖加载；使用依赖图管理 |
| **如何实现资源打包** 需要支持多种打包格式 | 支持 AssetBundle/Pak/自定义包格式等；实现增量打包；优化包体大小 |

| 典型资源类型 | 说明 |
|--------------|------|
| **3D 模型** | 模型加载、优化、格式管理 |
| **贴图** | 贴图加载、压缩、格式管理 |
| **材质** | 材质加载、参数管理、材质实例化 |
| **字体** | 字体加载、缓存、动态字体 |
| **骨骼** | 骨骼加载、骨骼动画管理 |
| **碰撞数据** | 碰撞数据加载、碰撞体类型管理 |
| **地图/关卡数据** | 关卡加载、关卡数据管理、流式加载 |

| 要点和思考方向 |
|----------------|
| 资源管理是游戏开发的基础 |
| 建立统一的资源规范提高可维护性 |
| 优化资源加载性能影响游戏体验 |
| 管理资源依赖关系避免加载错误 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「AssetBundle」「Addressables」「依赖」「打包」「热更新」 |
| **方法** | 让 AI 设计资源规范、依赖管理、打包流程 |
| **应用** | 资源架构、打包策略、热更新 |
| **提示词范例** | 「Unity 资源需 AssetBundle 或 Addressables，请设计资源命名规范、目录结构、依赖分析流程，以及增量打包与热更新的实现要点」 |

## 核心系统

| 维度 | 内容 |
|------|------|
| **作用** | 提供所有系统模块都可能用到的通用能力，是引擎的「基础库层」 |
| **应用场景** | 所有系统模块的基础支撑、通用功能实现、开发工具支持 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现核心系统功能** 需要提供各种核心系统功能 | 内存管理、单元测试框架、数学库、调试与日志系统、JSON/XML 数据解析；建立核心系统库；实现统一接口 |
| **如何建立日志规范** 需要统一的日志规范便于问题排查 | 统一的日志规范（等级、打点位置、输出格式）；为线上问题排查提供基础 |
| **如何提供数学库** 需要良好的数学库减少重复劳动 | 良好的数学库与工具；向量、矩阵、四元数、随机数等 |
| **如何实现单元测试** 需要单元测试框架降低演进风险 | 单元测试框架；支持自动化测试、持续集成 |

| 典型内容 | 说明 |
|----------|------|
| **内存管理** | 内存分配、回收、内存池 |
| **单元测试框架** | 单元测试、集成测试 |
| **数学库** | 向量、矩阵、四元数、随机数等 |
| **调试与日志系统** | 日志系统、调试工具、日志分级、日志过滤 |
| **JSON/XML 数据解析** | 数据解析、多格式支持 |

| 要点和思考方向 |
|----------------|
| 核心系统是引擎的基础库层 |
| 建立统一的规范和工具提高开发效率 |
| 优化核心系统性能影响整体性能 |
| 实现完善的测试框架保证代码质量 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「日志」「数学库」「单元测试」「内存池」「JSON」 |
| **方法** | 让 AI 设计日志规范、数学库封装、测试框架、内存管理 |
| **应用** | 核心库设计、日志系统、测试框架 |
| **提示词范例** | 「游戏需统一日志系统：分级、打点、输出格式，请设计 C# 日志接口、日志等级、以及线上问题排查的日志采集与过滤方案」 |

## 跨平台接口

| 维度 | 内容 |
|------|------|
| **作用** | 对操作系统和底层库进行封装，向上提供统一接口，让引擎逻辑尽量不关心具体平台差异 |
| **应用场景** | 多平台游戏开发、跨平台发布、底层库封装 |

| 问题 | 解决方向 |
|------|----------|
| **如何封装不同平台的功能** 需要封装不同平台的底层功能 | 文件系统、网络、容器/线程库、图形接口、物理接口；设计抽象层架构；实现平台适配层 |
| **如何实现接口统一** 需要让上层模块只依赖统一接口 | 让上层模块只依赖统一接口，可在项目后期更换/升级底层实现；设计统一接口；支持接口替换 |
| **如何简化多平台发布** 需要简化多平台发布流程 | 简化 PC/主机/移动端的差异处理；实现平台抽象、构建系统、自动化发布 |

| 典型封装方向 | 说明 |
|--------------|------|
| **文件系统** | 统一文件接口、异步 IO |
| **网络** | 统一网络接口、多协议支持 |
| **容器/线程库** | 统一容器接口、统一线程接口 |
| **图形接口** | 统一 DirectX/OpenGL/Vulkan/Metal 等 |
| **物理接口** | 对不同物理 SDK 的统一抽象 |

| 要点和思考方向 |
|----------------|
| 跨平台接口是引擎可移植性的基础 |
| 设计良好的抽象层提高可维护性 |
| 优化跨平台性能影响游戏体验 |
| 简化多平台发布流程提高开发效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明平台、需求；如「文件」「网络」「线程」「图形」「物理」「抽象层」 |
| **方法** | 让 AI 设计平台抽象层、接口统一、构建系统 |
| **应用** | 跨平台封装、接口设计、多平台构建 |
| **提示词范例** | 「引擎需跨平台：文件、网络、线程，请设计抽象接口与平台实现分离的 C# 架构，以及 PC/Android/iOS 的差异化处理策略」 |

## 游戏特有的系统

| 维度 | 内容 |
|------|------|
| **作用** | 针对不同游戏类型定制的系统，通常建立在引擎通用能力之上 |
| **应用场景** | 特定游戏类型开发、游戏特色功能实现、游戏玩法系统 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现游戏特有系统** 需要实现各种游戏特有系统 | 武器系统、载具系统、游戏特定渲染（水体、地形）、游戏摄像机、角色控制；基于引擎通用能力实现 |
| **如何沉淀可复用模块** 需要将通用部分抽象下沉 | 这些系统有明显的「品类特性」，但内部也会沉淀可复用的引擎模块；识别通用功能；抽象通用模块 |
| **如何形成游戏中间件** 需要在项目后期形成团队级中间件 | 建议在项目后期将通用部分抽象下沉，形成团队级的「游戏中间件」 |

| 举例 | 说明 |
|------|------|
| **武器系统** | 武器管理、属性、切换 |
| **载具系统** | 载具控制、物理、动画 |
| **游戏特定渲染** | 水的渲染、地形渲染 |
| **游戏摄像机** | 跟随相机、动画相机 |
| **角色控制** | 状态管理、移动、碰撞 |

| 要点和思考方向 |
|----------------|
| 游戏特有系统体现游戏特色 |
| 基于引擎通用能力实现 |
| 沉淀可复用模块提高效率 |
| 形成团队级中间件积累技术资产 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明游戏类型、需求；如「武器系统」「载具」「摄像机」「角色控制」「中间件」 |
| **方法** | 让 AI 设计游戏系统架构、可复用模块、抽象下沉 |
| **应用** | 游戏系统设计、模块抽象、中间件沉淀 |
| **提示词范例** | 「动作游戏需武器系统、角色控制，请设计基于引擎通用能力的模块划分，以及如何将通用部分抽象为团队级中间件的架构建议」 |

## 第三方接口 / SDK

| 维度 | 内容 |
|------|------|
| **作用** | 在操作系统之上，利用成熟的第三方库实现图形、物理、动画等基础功能服务 |
| **应用场景** | 图形渲染、物理模拟、动画处理、其他基础功能 |

| 问题 | 解决方向 |
|------|----------|
| **如何选择合适的 SDK** 需要选择合适的第三方 SDK | 评估性能、稳定性、许可协议、平台支持、社区活跃度 |
| **如何封装第三方 SDK** 需要合理封装第三方 SDK | 通过合理的封装，让第三方 SDK 可被安全升级或替换；实现抽象层；封装 SDK 接口 |
| **如何选择 SDK** 需要综合考虑多个因素选择 SDK | 综合考虑：性能、稳定性、许可协议、平台支持度与社区活跃度 |

| 类型 | SDK | 说明 |
|------|-----|------|
| **图形接口** | DirectX | Windows 平台图形，高性能渲染 |
| **图形接口** | OpenGL | 跨平台图形，广泛支持 |
| **图形接口** | Vulkan | 低开销 API，多线程支持，高性能 |
| **物理 SDK** | PhysX | NVIDIA 物理引擎，高性能 |
| **物理 SDK** | Havok | 专业物理引擎，商业授权 |
| **物理 SDK** | ODE | 开源物理引擎，免费使用 |
| **动画 SDK** | Granny | 专业动画引擎，商业授权 |

| 要点和思考方向 |
|----------------|
| 第三方 SDK 提供成熟的基础功能 |
| 合理封装提高可维护性和可替换性 |
| 综合考虑多个因素选择合适 SDK |
| 建立 SDK 管理规范提高开发效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「PhysX」「Havok」「DirectX」「Vulkan」「封装」「替换」 |
| **方法** | 让 AI 对比 SDK、设计封装层、给出选型建议 |
| **应用** | SDK 选型、封装设计、接口抽象 |
| **提示词范例** | 「游戏需物理引擎，请对比 PhysX、Havok、Bullet 的性能、许可、平台支持，并设计 C# 物理接口抽象层以便未来替换实现」 |
