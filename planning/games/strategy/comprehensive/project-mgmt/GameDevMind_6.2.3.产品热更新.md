## 为什么需要热更新？

| 维度 | 内容 |
|------|------|
| **作用** | 在不重新发布应用包的情况下，动态更新游戏内容、功能和数据 |
| **应用场景** | 内容和功能的更新；调整策划数据；修复产品bug；快速响应运营需求 |

| 运营需求 | 说明 |
|----------|------|
| **数据表热更** | 配置数据热更新 |
| **资源+代码热更** | 资源与代码动态更新 |

| 问题 | 解决方向 |
|------|----------|
| **如何应对快速变化的需求？** | 使用热更新快速响应需求；避免频繁发版；提升运营效率 |
| **如何避免用户流失？** | 通过热更避免用户流失；游戏用户不用下载安装新包；提升用户体验 |

| 要点和思考方向 |
|----------------|
| 热更新是网游产品快速迭代的基础能力 |
| 可以避免频繁发版导致的用户流失 |
| 需要支持差异化更新，不同渠道、版本可能需要不同的热更内容 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「热更新」「hotfix」「数据表热更」「资源+代码热更」 |
| **方法** | 让 AI 设计热更价值分析、差异化更新策略 |
| **应用** | 热更需求评估、渠道/版本差异化 |
| **提示词范例** | 「网游需热更新支持快速迭代，请分析数据表热更与资源+代码热更的应用场景，以及不同渠道、版本差异化更新的必要性」 |

## 热更新系统需要具备的功能

| 维度 | 内容 |
|------|------|
| **作用** | 提供完整的热更新功能，支持快速制作、发布和管理热更新 |
| **应用场景** | 热更新数据制作；热更新版本发布；热更新版本管理；热更新版本回滚 |

| 核心功能 | 说明 |
|----------|------|
| **差异化更新** | 支持不同渠道、版本的热更内容 |
| **快速制作和发布** | 提升运营效率 |
| **避免用户流失** | 游戏用户不用下载安装新包 |
| **一键生成可热更数据** | 简化数据制作流程 |
| **一键发布数据** | 自动上传、预热 |
| **可快速切换版本** | 支持版本回滚 |
| **版本管理** | 坐标：环境、平台、分支等 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现差异化更新？** | 支持不同渠道、版本的热更内容；实现差异化更新机制 |
| **如何提升制作和发布效率？** | 实现一键生成可热更数据；实现一键发布数据（自动上传和预热） |
| **如何实现版本管理？** | 实现多维度版本管理（环境、平台、分支）；支持快速切换版本和回滚 |

| 要点和思考方向 |
|----------------|
| 热更新系统要易用，降低运营成本 |
| 版本管理要灵活，支持多环境、多平台、多分支 |
| 需要支持快速回滚，应对线上问题 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「差异化更新」「一键生成」「一键发布」「版本回滚」「版本管理」 |
| **方法** | 让 AI 设计热更系统功能、发布流程、版本管理 |
| **应用** | 热更系统设计、发布自动化、版本切换 |
| **提示词范例** | 「热更新系统需差异化更新、一键生成可热更数据、一键发布（上传+预热）、版本回滚，请设计功能模块与多维度版本管理（环境、平台、分支）」 |

## 客户端方案

| 维度 | 内容 |
|------|------|
| **作用** | 在客户端实现热更新功能，支持动态加载资源和代码 |
| **应用场景** | 游戏客户端需要支持热更新；资源动态加载；代码动态更新 |

| 代码热更方案 | 候选 | 说明 |
|--------------|------|------|
| **Lua** | tolua、xlua | 使用脚本语言实现热更新 |
| **C#脚本** | — | 预先插入注入代码 |
| **C#热更** | HybridCLR、ILRuntime | 使用ILRuntime或HybridCLR实现C#热更新 |

| 资源更新功能 | 说明 |
|--------------|------|
| **指向更新后台** | 配置更新源 |
| **检查版本** | 版本检查 |
| **获取资源清单** | 获取资源清单 |
| **比较资源hash值** | 识别变化资源 |
| **同步/下载变化的资源** | 注意：不能直接覆盖原文件，需安全更新机制 |
| **更新时机** | 启动时检查更新；后台静默更新 |

| 依赖/对接系统 | 说明 |
|---------------|------|
| **HTTP网络系统** | 多线程并行下载 |
| **异常处理** | 网络下载异常处理、断点续传、重试机制 |
| **UI系统、元数据系统、脚本系统** | 对接系统 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现代码热更新？** | 使用Lua或ILRuntime/HybridCLR；根据项目需求选择合适的方案 |
| **如何实现资源更新？** | 实现版本检查和资源清单获取；比较资源hash值；安全下载和更新资源；合理选择更新时机 |
| **如何处理网络异常？** | 使用HTTP网络系统；实现断点续传、重试机制 |

| 要点和思考方向 |
|----------------|
| 客户端热更新需要支持资源更新和代码更新 |
| 脚本语言（如Lua）是常见的代码热更新方案 |
| 资源更新要支持差异更新，避免下载完整资源包 |
| 需要考虑网络异常、下载失败等异常情况 |
| 更新时机要合理，避免影响用户体验 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Lua」「xlua」「HybridCLR」「ILRuntime」「资源更新」「hash」「断点续传」 |
| **方法** | 让 AI 设计客户端热更方案、资源更新流程、异常处理 |
| **应用** | 代码热更选型、资源差异更新、下载与异常处理 |
| **提示词范例** | 「客户端热更新需支持 Lua/xlua 或 HybridCLR/ILRuntime 代码热更、资源差异更新（版本检查、hash 比较、安全下载），请设计更新流程与断点续传、重试机制」 |

## 服务端方案

| 维度 | 内容 |
|------|------|
| **作用** | 服务端热更新配置数据，支持动态调整游戏配置 |
| **应用场景** | 服务端需要支持配置数据热更新；游戏配置动态调整；不停服更新配置 |

| 数据特点 | 说明 |
|----------|------|
| **共用配置数据** | 服务端和客户端通常共用配置数据 |
| **不共用艺术和代码** | 服务端通常不需要共用艺术数据和代码 |
| **更新方式** | 数据包整体下载、释放，适合服务端数据特性 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现配置数据热更新？** | 实现配置数据包整体下载和释放；根据服务端数据特性选择合适的更新方式 |
| **如何保证数据一致性？** | 数据更新后同步更新内存中的元数据；更新被使用和引用的数据；更新由元数据衍生出来的数据 |

| 要点和思考方向 |
|----------------|
| 服务端热更新主要是配置数据更新 |
| 需要考虑数据更新后内存数据的同步更新 |
| 服务端数据更新要保证数据一致性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「配置数据热更」「数据包」「内存同步」「数据一致性」 |
| **方法** | 让 AI 设计服务端热更流程、数据加载与内存更新 |
| **应用** | 配置热更、不停服更新、数据一致性 |
| **提示词范例** | 「服务端需配置数据热更新，请设计数据包整体下载与释放、更新后内存元数据同步、被引用与衍生数据的更新逻辑，保证数据一致性」 |

## 大版本发布

| 维度 | 内容 |
|------|------|
| **作用** | 处理大版本更新时的热更新策略，确保版本兼容性 |
| **应用场景** | 大版本更新；版本兼容性处理；更新策略制定 |

| 兼容情况 | 处理策略 |
|----------|----------|
| **老版本兼容新热更** | 直接制作、发布、应用最新内容的热更 |
| **老版本不兼容新热更** | 只针对最新包开启热更检查；3天内非强制更新提醒；3天后强制更新提醒 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理版本兼容性？** | 评估老版本客户端兼容性；根据兼容情况选择相应策略 |
| **如何制定更新策略？** | 制定合理的更新策略；平衡用户体验和版本统一；谨慎使用强制更新，避免用户流失 |

| 要点和思考方向 |
|----------------|
