## 反向代理（Reverse Proxy）

| 维度 | 内容 |
|------|------|
| **作用** | 对客户端请求进行引导和调度；隐藏后端实际业务地址和端口，降低被攻击风险；控制负载 |
| **应用场景** | 作为游戏服务的统一入口；作为网络跳板（如香港跳板访问境外主机）；负载均衡；安全防护 |

| 候选方案 | 说明 |
|----------|------|
| **Nginx** | 高性能、轻量级，广泛使用 |
| **HAProxy** | 专业的负载均衡器，功能强大 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决单点故障问题？** | 配置主备模式；使用Keepalived实现自动故障转移；配置健康检查机制 |
| **如何解决性能瓶颈问题？** | 选择高性能的反向代理方案；优化配置参数；使用缓存机制提升性能 |
| **如何解决配置复杂问题？** | 使用配置模板简化配置；使用版本控制管理配置；实现自动化配置管理 |

| 要点和思考方向 |
|----------------|
| Nginx：高性能、轻量级，广泛使用 |
| HAProxy：专业的负载均衡器，功能强大 |
| 反向代理是服务架构中的重要组件，需要合理配置 |
| 可以隐藏后端服务，提升安全性 |
| 可以实现负载均衡，提升服务可用性 |
| 配置高可用方案，避免单点故障 |
| 根据业务需求选择合适的反向代理方案 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Nginx」「HAProxy」「反向代理」「Keepalived」「负载均衡」 |
| **方法** | 让 AI 设计配置、高可用方案、健康检查 |
| **应用** | 反向代理配置、主备切换、负载均衡 |
| **提示词范例** | 「游戏服务需 Nginx 反向代理，请设计主备+Keepalived 高可用配置、健康检查与故障转移，以及 Nginx 与 HAProxy 的选型对比」 |

## 云主机

| 维度 | 内容 |
|------|------|
| **作用** | 提供虚拟化的计算资源，是游戏服务器的基础运行环境 |
| **应用场景** | 游戏服务器部署；开发测试环境；临时资源扩展；云上服务运行 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决配置错误问题？** 快照：出现问题后可以快速回滚/恢复 | 定期创建快照；配置错误时快速回滚；在测试环境验证配置 |
| **如何解决系统故障问题？** 镜像：可用于快速迁移服务或克隆新环境 | 创建系统镜像；快速迁移服务到新主机；使用镜像快速恢复服务 |
| **如何解决数据丢失问题？** | 定期创建快照；配置自动备份机制；实现多地备份 |
| **如何解决安全风险问题？** | 合理开放端口，配置防火墙和安全组；使用云服务商提供的安全产品（DDoS防护、入侵检测等）；定期安全审计 |
| **如何解决网络问题？** | 配置流量控制；限制带宽使用；监控网络流量，避免成本超支 |

|----------------|
| 快照是重要的备份和恢复手段，需要定期创建 |
| 镜像可以快速复制环境，提升部署效率 |
| 安全配置很重要，需要遵循最小权限原则 |
| 网络流量控制可以避免成本超支 |
| 定期创建快照和镜像，确保快速恢复 |
| 配置完善的安全策略，使用安全产品 |
| 监控网络流量，控制成本 |

| 问题 | 解决方向 |
|------|----------|
| **如何选择IDC？** | 评估IDC基础设施和运维能力；评估网络质量和安全措施；考虑地理位置和成本 |
| **如何管理IDC资源？** | 建立专业运维团队；实现资源监控和管理；建立完善的运维流程 |
| **如何控制成本？** | 评估IDC成本，优化资源配置；考虑混合云方案降低成本；平衡成本和控制权 |
| **如何保证安全性？** | 实现物理、网络、数据安全；定期进行安全审计；建立安全管理制度 |

| 要点和思考方向 |
|----------------|
| IDC适合对数据安全、网络质量要求极高的场景 |
| 需要专业的运维团队和基础设施 |
| 目前中小团队基本不用，更多使用云服务 |
| 成本较高，但可以提供更高的控制权和安全性 |
