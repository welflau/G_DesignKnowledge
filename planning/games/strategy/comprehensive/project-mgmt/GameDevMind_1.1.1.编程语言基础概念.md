## 常量

| 维度 | 内容 |
|------|------|
| **作用** | 告诉编译器（实际上是为了告诉其他开发者），这是个不能去修改的量 |
| **应用场景** | 定义配置值（如游戏版本号、最大玩家数）；魔法数字（如圆周率、重力加速度）；枚举值（如游戏状态、错误码）；需要保持不变的数据 |

| 问题 | 解决方向 |
|------|----------|
| **魔法数字问题**：代码中直接使用数字，难以理解和维护 | 使用常量替代魔法数字；给常量起有意义的名称；集中管理常量定义 |
| **常量误修改**：常量被意外修改，导致程序错误 | 使用语言提供的常量机制（如 const、final）；编译器会阻止对常量的修改；代码审查时注意常量使用 |

| 要点和思考方向 |
|----------------|
| 使用常量提高代码可读性和可维护性 |
| 集中管理常量，便于统一修改 |
| 给常量起有意义的名称，避免魔法数字 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 让 AI 将魔法数字提取为常量时，说明用途和命名意图，如「把这段代码中的 60、100 提取为常量，分别表示最大玩家数和超时秒数」 |
| **方法** | 提供业务上下文，AI 才能给出有意义的常量名；要求按模块或功能集中定义常量类/文件 |
| **应用** | 用 AI 批量扫描代码中的魔法数字并建议常量名；生成常量定义文件骨架 |

## 变量

| 维度 | 内容 |
|------|------|
| **作用** | 存储数据值，在程序运行过程中可以修改 |
| **应用场景** | 所有需要存储和操作数据的场景 |
| **作用域类型** | 局部变量（函数内部，生命周期短）；静态变量（程序运行期间保持存在，只初始化一次）；全局变量（整个程序可访问） |

| 问题 | 解决方向 |
|------|----------|
| **作用域混乱**：导致变量访问错误，如访问未定义的变量或访问了错误的变量 | 明确变量的作用域范围；避免使用过长的变量名或容易混淆的变量名；使用命名空间或类来组织变量；避免全局变量污染 |
| **生命周期管理不当**：导致内存问题，如使用已释放的变量、内存泄漏 | 理解不同作用域变量的生命周期；及时释放不再使用的资源；使用智能指针（C++）或自动垃圾回收（Java、C#）；避免悬空指针和野指针 |
| **变量命名不规范**：导致代码可读性差 | 使用有意义的变量名；遵循命名规范（如驼峰命名、下划线命名）；避免使用单字母变量名（除了循环变量） |

| 要点和思考方向 |
|----------------|
| 理解变量的作用域和生命周期，避免访问错误 |
| 合理使用局部变量、静态变量和全局变量 |
| 避免全局变量污染，优先使用局部变量 |
| 注意内存管理，及时释放不再使用的资源 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 描述变量用途和生命周期，如「需要一个在函数内累加计数的局部变量」「需要一个进程级单例缓存」 |
| **方法** | 明确指定语言和内存模型（如 C++ 栈/堆、GC 语言），AI 才能给出正确的变量声明和释放建议 |
| **应用** | 让 AI 重构作用域混乱的代码；检查变量命名是否符合规范；生成符合项目风格的变量声明 |

## 数据类型

| 维度 | 内容 |
|------|------|
| **常用数据类型** | int、long、float、char、string、bool、数组、enum |
| **大整数** | 当原生整数类型无法满足需求时使用（如玩家的经验值、金币等） |

| 问题 | 解决方向 |
|------|----------|
| **数据范围溢出**：运算中数据超出范围，会造成异常 | 选择合适的数据类型，或使用大整数类型 |
| **数据精度问题**：画面（3D 的面）有闪烁时，考虑是精度问题 | 根据精度需求选择 float 或 double，注意浮点数精度限制 |
| **数据类型长度**：计算数据存储长度或内存中的长度时需考虑类型长度 | 了解不同平台下数据类型的实际长度，使用 sizeof 等工具 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明数据范围和精度需求，如「玩家经验值可能超过 int 上限，需要大整数」「3D 坐标需要高精度，避免 Z-fighting」 |
| **方法** | 提供平台信息（32/64 位、目标语言版本），AI 才能推荐合适的类型（如 `int64_t`、`BigInt`、`decimal`） |
| **应用** | 让 AI 分析溢出风险并建议类型；生成类型别名和序列化/反序列化代码；解释不同平台下 sizeof 差异 |

## 运算符

| 维度 | 内容 |
|------|------|
| **逻辑运算** | 用于条件判断 |
| **位运算** | 用于底层操作、性能优化（速度快、省内存） |
| **数学运算** | 基本算术操作 |

| 问题 | 解决方向 |
|------|----------|
| **运算符优先级混淆**：复杂的表达式可能因为优先级理解错误导致逻辑错误 | 尽量用括号来设定优先级，简单、可靠；查阅运算符优先级表；将复杂表达式拆分为多个简单表达式；代码审查时注意复杂表达式 |
| **位运算错误**：位运算使用不当，导致逻辑错误或性能问题 | 理解位运算的含义和用途；注意位运算的优先级；使用位运算时要考虑可读性；在性能关键路径上使用位运算 |
| **除零错误**：除法运算时除数为零，导致程序崩溃 | 在除法运算前检查除数是否为零；使用异常处理机制；提供默认值或错误处理 |

| 要点和思考方向 |
|----------------|
| 运算符优先级要明确，使用括号提高可读性 |
| 位运算速度快、省内存，适合底层操作和性能优化 |
| 注意除零错误，做好边界检查 |
| 复杂表达式要拆分，提高可读性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 描述运算目的和边界条件，如「实现权限位掩码的与/或运算」「除法前需要检查除零」 |
| **方法** | 对复杂表达式，让 AI 加括号或拆成多行并注释；对位运算，要求 AI 解释含义和用途 |
| **应用** | 用 AI 重写易混淆的表达式并加括号；生成带除零检查的运算代码；解释位运算在特定场景的用法 |

## 函数

| 维度 | 内容 |
|------|------|
| **参数** | 函数输入 |
| **返回值** | 函数输出 |
| **构造函数、析构函数** | 对象的创建和销毁 |

| 问题 | 解决方向 |
|------|----------|
| **内存泄漏**：构造函数和析构函数使用不当导致资源未正确释放 | 保证内存的正确分配和回收；使用 RAII（资源获取即初始化）原则；使用智能指针（C++）或自动垃圾回收；在析构函数中释放所有分配的资源 |
| **堆栈溢出**：递归调用过深导致堆栈溢出，程序崩溃 | 尽量少用递归，堆栈深度是有限的；考虑使用迭代替代递归；使用尾递归优化（如果语言支持）；限制递归深度，添加深度检查 |
| **参数传递错误**：参数传递方式不当，导致性能问题或逻辑错误 | 理解不同参数传递方式的区别；根据数据类型和需求选择合适的传递方式；大对象使用引用或指针传递，避免拷贝开销 |
| **返回值处理不当**：忽略函数返回值，导致错误未被处理 | 检查并处理所有函数返回值；使用错误码或异常机制；对于关键函数，强制检查返回值 |

| 要点和思考方向 |
|----------------|
| 函数是代码复用的基本单元，要设计良好的函数接口 |
| 注意内存管理，保证资源的正确分配和回收 |
| 避免过深的递归调用，考虑使用迭代 |
| 理解参数传递方式，选择合适的传递方式 |
| 处理函数返回值，避免忽略错误 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明参数语义和传递方式，如「大对象用 const 引用，小类型用值传递」「返回值表示错误码，0 为成功」 |
| **方法** | 提供函数签名或接口文档，让 AI 生成实现；要求 AI 将递归改为迭代时说明递归逻辑 |
| **应用** | 用 AI 设计函数签名和参数；生成 RAII/析构逻辑；检查返回值是否被正确处理；将递归改写为尾递归或迭代 |

## 流程控制

| 维度 | 内容 |
|------|------|
| **条件语句** | if、switch |
| **循环语句** | for、loop、while |

| 问题 | 解决方向 |
|------|----------|
| **switch 语句遗漏 break**：忘记 break 会导致意外的 fall-through 行为，执行多个 case 分支 | 不要遗忘 break，或明确使用 fall-through 注释；使用编译器警告（如 -Wswitch-fallthrough）；在需要 fall-through 的地方明确注释 |
| **无限循环**：循环条件设置错误，导致无限循环，程序卡死 | 确保循环条件能够终止；在循环体内修改循环变量；使用 break 或 return 提前退出循环；添加循环次数限制（如果适用） |
| **循环性能问题**：循环内部执行耗时操作，导致性能问题 | 优化循环内部操作；减少循环嵌套；使用更高效的循环方式（如范围 for 循环）；考虑并行化处理（如果适用） |
| **条件判断错误**：条件判断逻辑错误，导致程序行为异常 | 仔细检查条件判断逻辑；使用括号明确优先级；使用布尔变量简化复杂条件；编写单元测试验证逻辑 |

| 要点和思考方向 |
|----------------|
| 流程控制是程序逻辑的基础，要确保逻辑正确 |
| 注意 switch 语句的 break，避免 fall-through |
| 确保循环能够终止，避免无限循环 |
| 优化循环性能，减少不必要的操作 |
| 使用清晰的条件判断，提高代码可读性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 描述业务逻辑和终止条件，如「遍历玩家列表直到找到目标或到达末尾」「根据状态码分支，每个 case 必须 break」 |
| **方法** | 用自然语言描述流程，让 AI 生成 if/switch/循环；要求 AI 在 switch 中显式标注 fall-through 或加 break |
| **应用** | 用 AI 将伪代码或需求转为流程控制代码；检查循环是否有终止条件；优化嵌套循环结构；生成带边界检查的条件判断 |

## 面向对象

| 维度 | 内容 |
|------|------|
| **核心概念** | 类、对象、继承、多态、抽象、虚接口 |
| **重载与覆盖** | 两个或两个以上的函数，具有相同的函数名，但形参的个数或类型不同，编译器根据实参和形参的类型及个数进行最佳匹配 |

| 问题 | 解决方向 |
|------|----------|
| **重载函数参数冲突**：重载函数的参数类型和参数个数一定要不同，否则会导致编译错误 | 确保重载函数的参数签名有明确区别；参数类型、个数或顺序要不同；返回值类型不能作为重载的依据 |
| **虚析构问题**：基类指针删除派生类对象时，如果基类析构函数不是虚函数，可能导致资源泄漏 | 多态基类应使用虚析构函数；如果基类有虚函数，析构函数也应该是虚函数；确保派生类资源正确释放 |
| **继承层次过深**：导致代码复杂、难以维护 | 限制继承层次深度（通常不超过 3-4 层）；优先使用组合而非继承；使用接口而非具体类继承 |
| **多态性能开销**：虚函数调用有性能开销，在性能关键路径上可能影响性能 | 在性能关键路径上减少虚函数调用；使用内联函数（如果可能）；考虑使用模板或其他技术 |

| 要点和思考方向 |
|----------------|
| 面向对象提供代码组织和复用的机制 |
| 理解继承、多态、封装等核心概念 |
| 合理使用继承，避免过深的继承层次 |
| 多态基类应使用虚析构函数 |
| 注意多态的性能开销，在关键路径上优化 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明继承关系和设计意图，如「基类需要虚析构，因为会通过基类指针删除派生类」「优先用组合替代继承」 |
| **方法** | 给出类图或 UML 片段，让 AI 生成类定义；要求 AI 检查虚析构、重载冲突、继承层次深度 |
| **应用** | 用 AI 设计类层次和接口；生成多态基类与派生类；重构过深继承为组合；分析虚函数调用热点 |

## 断言

| 维度 | 内容 |
|------|------|
| **作用** | 在一段代码的前面做出一个假定，预判某些值一定在你设定的范围，如果没有在这个范围，就是错误的情况。它能帮你快速发现代码中的一些问题 |
| **应用场景** | 单元测试；函数参数的检查；某些结果的检查；调试 |

| 问题 | 解决方向 |
|------|----------|
| **断言在生产环境的影响**：断言失败可能导致程序崩溃，影响生产环境稳定性 | 在生产环境中禁用断言（使用编译选项）；使用异常处理替代断言（在生产环境）；区分开发断言和生产断言 |
| **断言使用不当**：如检查用户输入（应该用异常处理） | 断言用于检查不应该发生的内部错误；用户输入和外部数据用异常处理；明确断言和异常的使用场景 |

| 要点和思考方向 |
|----------------|
| 断言用于开发阶段发现内部错误，不应该用于处理用户输入 |
| 在生产环境中通常禁用断言，避免影响稳定性 |
| 区分断言和异常的使用场景 |
| 使用断言提高代码质量和可维护性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 明确断言与异常的分工，如「对内部不变式用 assert，对用户输入用异常」「断言仅在 Debug 构建生效」 |
| **方法** | 说明前置条件、后置条件、不变量，让 AI 在合适位置插入断言；要求区分开发断言与生产错误处理 |
| **应用** | 用 AI 在关键路径添加断言；检查断言是否误用于用户输入；生成带条件编译的断言宏（如 NDEBUG） |

## 异常

| 维度 | 内容 |
|------|------|
| **作用** | 捕获异常并且做出明确处理，能避免发生异常后中断后续代码，造成不好预测的情况 |
| **应用场景** | 处理运行时错误（如文件不存在、网络错误）；处理用户输入错误；处理资源获取失败（如内存分配失败）；处理业务逻辑异常 |

| 问题 | 解决方向 |
|------|----------|
| **异常未被捕获**：导致程序崩溃，用户体验差 | 在适当的地方捕获异常；使用全局异常处理器（如果语言支持）；记录异常信息，便于调试；提供友好的错误提示 |
| **异常处理不当导致资源泄漏**：如文件未关闭、内存未释放 | 使用 try-finally 或 using 语句确保资源释放；使用 RAII 原则（C++）；在异常处理中也要释放资源；使用智能指针或自动资源管理 |
| **异常性能开销**：异常处理有性能开销，过度使用可能影响性能 | 异常用于异常情况，不用于正常流程控制；在性能关键路径上减少异常使用；使用错误码替代异常（如果适用） |
| **异常信息不足**：难以定位问题 | 提供详细的异常信息；包含上下文信息（如函数名、参数值）；使用异常链（如果语言支持）；记录异常堆栈信息 |

| 要点和思考方向 |
|----------------|
| 异常用于处理异常情况，不用于正常流程控制 |
| 确保异常被正确捕获和处理 |
| 在异常处理中也要释放资源，避免资源泄漏 |
| 提供详细的异常信息，便于调试 |
| 注意异常的性能开销，在关键路径上优化 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明异常来源和期望处理方式，如「文件打开失败应抛出并带路径信息」「在 catch 中必须关闭已打开的文件句柄」 |

## 宏

| 维度 | 内容 |
|------|------|
| **作用** | 在编译时进行文本替换，实现代码生成和条件编译 |
| **应用场景** | 代码功能的开关（如基于版本或平台）；定义常量或函数；条件编译（如不同平台的代码）；代码生成和模板 |

| 问题 | 解决方向 |
|------|----------|
| **宏定义错误**：导致编译错误或逻辑错误 | 仔细检查宏定义，注意括号和优先级；使用内联函数替代简单宏（如果可能）；使用 constexpr（C++）替代宏常量 |
| **宏调试困难**：宏在预处理阶段展开，调试困难 | 使用编译器选项查看宏展开结果；避免复杂的宏定义；使用函数替代复杂宏 |
| **宏命名冲突**：导致意外的文本替换 | 使用命名空间或前缀避免冲突；使用全大写命名宏；及时取消宏定义（#undef） |

| 要点和思考方向 |
|----------------|

| 使用宏实现条件编译和代码生成 |
| 避免复杂的宏定义，优先使用函数 |
| 注意宏命名冲突，使用合适的命名规范 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明宏的用途和展开环境，如「需要跨平台的日志开关」「用宏生成重复的序列化代码，注意括号和副作用」 |
| **方法** | 提供目标平台和编译器，让 AI 生成条件编译宏；对复杂宏要求 AI 给出展开示例并检查优先级 |
| **应用** | 用 AI 生成条件编译块（#ifdef）；将易出错的宏改写为 inline/constexpr；检查宏命名是否与现有符号冲突 |

## 名字空间

| 维度 | 内容 |
|------|------|
| **作用** | 组织代码模块，避免命名冲突，提高代码可维护性 |
| **应用场景** | 大型项目的代码组织；库和框架的开发；模块化开发；避免命名冲突 |

### 价值

| 价值 |
|------|
| 使用名字空间可以更好的组织代码模块 |
| 让交付的代码模块被作为一个整体去使用 |
| 提高代码的可读性和可维护性 |

| 问题 | 解决方向 |
|------|----------|
| **名字冲突**：不同模块的同名符号冲突，导致编译错误或链接错误 | 合理使用名字空间进行模块划分；使用嵌套名字空间组织复杂项目；使用 using 声明或 using 指令（但要谨慎）；使用别名简化长名字空间 |
| **名字空间使用不当**：导致代码混乱 | 避免过度使用 using namespace；使用 using 声明而非 using 指令；保持名字空间层次清晰；遵循项目的命名规范 |

| 要点和思考方向 |
|----------------|
| 名字空间用于组织代码，避免命名冲突 |
| 合理划分名字空间，保持层次清晰 |
| 避免过度使用 using namespace |
| 遵循命名规范，提高代码可维护性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明模块划分和命名规范，如「按功能划分 namespace，如 Game::Network、Game::Render」「避免在头文件中 using namespace」 |
| **方法** | 给出项目结构或模块列表，让 AI 设计名字空间层次；要求 AI 在 .cpp 中适度使用 using 声明 |
| **应用** | 用 AI 重构分散代码到合适名字空间；生成 namespace 和别名；检查 using namespace 的使用范围；解决命名冲突 |
