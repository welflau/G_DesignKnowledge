## 网络

| 维度 | 内容 |
|------|------|
| **作用** | 优化网络消息传输，减少流量和延迟 |
| **应用场景** | 所有网络通信、客户端-服务端通信、服务端间通信 |
| **做什么的？** | 优化网络消息传输，减少流量和延迟 |
| **在哪用？** | 所有网络通信 |

| 问题 | 解决方向 |
|------|----------|
| **如何减少网络消息的数据量？** 网络消息数据量大，影响性能和成本 | 消息压缩（ProtoBuf、gzip）；优化消息结构；使用二进制协议；实现消息合并；减少冗余数据 |
| **如何优化网络性能？** 网络延迟和带宽影响性能 | 使用内网通信；优化网络路径；使用 CDN 加速；实现消息批处理；优化网络协议 |
| **网络架构优化** 网络架构可能影响性能 | 检查服务间访问是否走内网；使用内网通信减少延迟；优化网络拓扑；使用负载均衡；实现网络监控 |

| 压缩方案 | 优点 | 缺点 |
|----------|------|------|
| **ProtoBuf** | 体积小、序列化快、跨语言 | 需要定义消息结构、需要转换工作 |
| **gzip** | 实现简单、兼容性好 | 压缩率相对较低 |

| 优化意义 | 说明 |
|----------|------|
| **运营更省钱** | 更省流量（特别是全球加速场景） |
| **玩家体验更好** | 玩家更省流量、前后端交互更流畅、手机耗电更慢 |

| 要点和思考方向 |
|----------------|
| 消息压缩可以显著减少流量和成本 |
| 使用内网通信提高性能 |
| 优化消息结构减少数据量 |
| 监控网络性能指标 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明协议、压缩需求；如「ProtoBuf」「gzip」「消息压缩」「内网」 |
| **方法** | 让 AI 对比压缩方案、设计消息结构、给出内网优化建议 |
| **应用** | 消息压缩选型、协议优化、网络架构优化 |
| **提示词范例** | 「游戏服消息需压缩，请对比 ProtoBuf 与 JSON+gzip 的体积、序列化性能，并给出 C# 或 Go 下 ProtoBuf 消息定义和批量发送的示例」 |

## 负载

| 维度 | 内容 |
|------|------|
| **作用** | 优化服务器负载，提高处理能力 |
| **应用场景** | 所有服务端系统、高并发场景、性能优化 |
| **做什么的？** | 优化服务器负载，提高处理能力 |
| **在哪用？** | 所有服务端系统 |

| 问题 | 解决方向 |
|------|----------|
| **如何提高服务器的处理能力？** 服务器处理能力不足 | 性能记录：统计响应时间（最大、最小、平均）；性能分析：识别瓶颈、制定优化方案；性能优化：分布式、多进程、分离模块 |
| **如何充分利用CPU多核？** CPU 多核未充分利用 | 使用多进程/多线程；实现负载均衡；优化任务分配；使用异步处理；避免锁竞争 |
| **负载监控和优化** 需要监控和优化负载 | 检查 CPU 多核使用、CPU 使用率；监控 CPU、内存、网络 IO、磁盘 IO；设置负载告警 |

| 优化方法 | 说明 |
|----------|------|
| **分布式** | 将服务分布到多台服务器；提高处理能力、可用性；实现负载均衡 |
| **多进程** | 充分利用多核 CPU；提高并发处理能力；隔离进程故障 |
| **分离模块** | 将不同功能分离到不同模块；提高模块独立性；便于扩展和维护 |

| 要点和思考方向 |
|----------------|
| 性能记录和分析是优化的基础 |
| 分布式、多进程、分离模块是提高处理能力的有效方法 |
| 充分利用 CPU 多核提高性能 |
| 建立完善的监控体系 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明规模、瓶颈；如「分布式」「多进程」「多核」「分离模块」 |
| **方法** | 让 AI 设计负载均衡、多进程架构、模块拆分方案 |
| **应用** | 负载优化、多核利用、模块拆分 |
| **提示词范例** | 「游戏服 CPU 单核跑满，请说明多进程、多线程、Actor 模型的适用场景，以及 Go 或 Node.js 下如何充分利用多核」 |

## 并发

| 维度 | 内容 |
|------|------|
| **作用** | 处理并发操作，避免数据竞争 |
| **应用场景** | 需要处理并发请求的系统、高并发场景、多线程/多进程系统 |
| **做什么的？** | 处理并发操作，避免数据竞争 |
| **在哪用？** | 需要处理并发请求的系统 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理并发异步操作？** 并发异步操作可能导致数据竞争和时序问题 | 一次执行一个事务/任务，避免 IO 读写交叉使用；使用锁机制、消息队列、Actor 模型；实现事务管理；使用无锁数据结构 |
| **如何避免IO读写的交叉使用？** IO 读写交叉使用可能导致数据不一致 | 使用队列串行化 IO 操作；使用事务保证原子性；实现读写分离；使用连接池管理；避免并发 IO 冲突 |
| **并发性能优化** 并发处理可能影响性能 | 减少锁竞争；使用无锁编程；优化锁粒度；使用读写锁；实现并发控制策略 |

| 要点和思考方向 |
|----------------|
| 并发异步操作需要避免数据竞争 |
| 使用队列串行化 IO 操作避免交叉使用 |
| 合理使用锁机制 |
| 优化并发性能 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明并发场景；如「锁」「消息队列」「Actor」「IO 串行化」 |
| **方法** | 让 AI 设计并发控制、IO 串行化、避免数据竞争方案 |
| **应用** | 并发控制设计、锁优化、Actor 实现 |
| **提示词范例** | 「高并发下数据库读写交叉导致数据不一致，请设计 IO 操作串行化方案，以及消息队列与 Actor 模型的取舍」 |

## 压力测试

| 维度 | 内容 |
|------|------|
| **作用** | 测试服务器在高负载下的表现 |
| **应用场景** | 服务器上线前、性能优化阶段、容量规划、性能评估 |
| **做什么的？** | 测试服务器在高负载下的表现 |
| **在哪用？** | 服务器上线前和性能优化阶段 |

| 问题 | 解决方向 |
|------|----------|
| **如何模拟真实玩家的访问？** 需要模拟真实玩家行为进行压力测试 | 模拟机器人：录制重放（真实反映行为，需大量录制数据）、机器学习（可生成大量数据，需训练模型）；使用压力测试工具；实现自定义测试脚本；模拟不同场景 |
| **如何评估服务器的性能？** 需要量化评估服务器性能 | 监控 RT、TPS、资源使用率（CPU、内存、网络、磁盘）、错误率；分析性能瓶颈；建立性能基线 |
| **压力测试工具和方法** 需要选择合适的压力测试工具和方法 | 第三方测试服务（专业全面，成本较高）；开源工具（JMeter、Locust）；自定义压力测试框架；建立压力测试流程 |

| 模拟方式 | 优点 | 缺点 |
|----------|------|------|
| **录制重放** | 真实反映玩家行为 | 需要大量录制数据 |
| **机器学习** | 可以生成大量测试数据 | 需要训练模型 |

| 要点和思考方向 |
|----------------|
| 压力测试是评估服务器性能的重要手段 |
| 模拟真实玩家行为很重要 |
| 建立完善的性能监控体系 |
| 定期进行压力测试 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明工具、场景；如「JMeter」「Locust」「录制重放」「压测脚本」 |
| **方法** | 让 AI 生成压测脚本、设计压测场景、给出工具选型建议 |
| **应用** | 压测脚本编写、场景设计、工具选型 |
| **提示词范例** | 「游戏服需压测，请对比 JMeter 与 Locust 的适用场景，并给出 Locust 模拟玩家登录、进入战斗、发送消息的 Python 脚本示例」 |

## 数据库优化

| 维度 | 内容 |
|------|------|
| **作用** | 优化数据库访问性能 |
| **应用场景** | 所有需要数据库的系统、高并发数据库访问、性能优化 |
| **做什么的？** | 优化数据库访问性能 |
| **在哪用？** | 所有需要数据库的系统 |

| 问题 | 解决方向 |
|------|----------|
| **如何优化数据库响应时间？** 数据库响应时间过长 | 使用缓存（Redis、多级缓存）；批量读写；异步 IO；MySQL 优化（索引、数据结构）；优化 SQL；读写分离；分库分表 |
| **如何优化网络延迟？** 数据库网络延迟影响性能 | 使用内网连接；使用数据库连接池；优化网络路径；使用就近数据库；实现数据库代理 |
| **如何设计连接池？** 连接池设计不合理影响性能 | 池子太大增加数据库压力，池子太小让业务堆积；根据并发量设置连接数；监控连接使用；实现动态调整；处理连接泄漏；设置连接超时 |
| **数据库集群和扩展** 单数据库性能不足 | 主从复制、读写分离、分片存储；使用数据库代理；实现负载均衡；监控集群状态 |
| **不同数据库的优化** 不同数据库有不同的优化方法 | MongoDB：优化索引、查询、分片、文档结构；Redis：优化数据结构、使用管道、实现集群、优化内存 |

| 连接池检查 | 说明 |
|------------|------|
| **池子太大** | 会增加数据库压力 |
| **池子太小** | 会让业务堆积，处理延时 |

| 要点和思考方向 |
|----------------|
| 使用缓存、批量读写、异步 IO 优化响应时间 |
| 合理设计连接池大小 |
| 使用集群提高性能 |
| 针对不同数据库采用不同的优化策略 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明数据库类型、瓶颈；如「连接池」「Redis」「读写分离」「分库分表」 |
| **方法** | 让 AI 设计连接池、缓存策略、分库分表方案 |
| **应用** | 连接池设计、缓存设计、数据库架构优化 |
| **提示词范例** | 「MySQL 连接池如何设置大小？请说明根据并发量、RT 计算连接数的公式，以及连接泄漏检测和动态调整的实现思路」 |
