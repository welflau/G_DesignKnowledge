## 数据对象

| 维度 | 内容 |
|------|------|
| **作用** | 提供静态数据存储 |
| **应用场景** | 存储游戏热更数据等静态配置数据；游戏配置文件存储；静态资源元数据存储；不需要频繁修改的数据存储 |

| 问题 | 解决方向 |
|------|----------|
| **如何选择合适的存储方式？** 对象存储（OSS/S3）、文件存储、数据库存储 | 根据数据大小和访问频率选择；考虑存储成本和访问速度；考虑数据安全性和备份需求 |
| **如何保证数据一致性？** | 使用版本号管理数据；实现数据更新机制；确保数据一致性 |
| **如何优化访问性能？** | 使用CDN加速访问；实现本地缓存机制；优化数据加载策略 |

| 要点和思考方向 |
|----------------|
| 主要用于存储不需要频繁修改的静态数据 |
| 适合游戏配置、热更资源等场景 |
| 选择合适的存储方式（对象存储、文件存储等） |
| 考虑数据版本管理和一致性 |
| 优化访问性能，使用CDN和缓存 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「OSS」「S3」「对象存储」「热更」「版本号」 |
| **方法** | 让 AI 设计存储选型、版本管理、访问优化 |
| **应用** | 静态数据存储、热更配置、CDN 配合 |
| **提示词范例** | 「游戏热更数据等静态配置需存储，请设计 OSS/S3 对象存储与 CDN 的配合方案、数据版本管理与一致性机制，以及访问性能优化策略」 |

## 缓存数据库

| 维度 | 内容 |
|------|------|
| **作用** | 让数据操作更高效；让内存使用更可控；提供快速数据访问 |
| **应用场景** | 对数据IO性能要求比较高的业务；热点数据存储；频繁读写的数据；会话数据存储；临时数据存储 |
| **特点** | 将数据存储到内存，读写速度远快于磁盘数据库 |

| 候选方案 | 说明 |
|----------|------|
| **Redis** | 功能丰富，支持多种数据结构，支持持久化 |
| **MemCache** | 简单高效，纯内存缓存 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决内存容量限制？** | 使用集群模式扩展容量；实现数据分片；合理规划内存使用 |
| **如何选择持久化策略？** 持久化（RDB、AOF）或仅内存 | 根据业务需求选择；配置RDB或AOF持久化；平衡性能和数据安全 |
| **如何保证高可用性？** | 配置主从复制；使用哨兵模式实现自动故障转移；使用集群模式提升可用性 |
| **如何选择缓存数据库？** | 根据业务需求选择Redis或MemCache；考虑功能需求和性能要求 |

| 要点和思考方向 |
|----------------|
| Redis：功能丰富，支持多种数据结构，支持持久化 |
| MemCache：简单高效，纯内存缓存 |
| 缓存数据库适合热点数据、频繁读写的数据 |
| 需要考虑集群配置以支持高并发和高可用 |
| 根据业务需求选择合适的持久化策略 |
| 配置高可用性方案，确保服务稳定性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Redis」「MemCache」「持久化」「RDB」「AOF」「哨兵」「集群」 |
| **方法** | 让 AI 设计缓存架构、持久化策略、高可用配置 |
| **应用** | 热点数据、会话存储、高可用缓存 |
| **提示词范例** | 「游戏需缓存：热点数据、会话，请对比 Redis 与 MemCache，设计持久化策略（RDB/AOF）、主从与哨兵高可用配置，以及集群分片方案」 |

## MongoDB

| 维度 | 内容 |
|------|------|
| **类型** | 非关系型数据库（NoSQL） |
| **作用** | 提供灵活的文档存储，支持非结构化或半结构化数据 |
| **应用场景** | 游戏日志存储；玩家行为数据存储；灵活数据结构需求；水平扩展需求 |

| 问题 | 解决方向 |
|------|----------|
| **如何保证数据一致性？** | 配置副本集、分片集群；使用分片集群实现水平扩展；处理最终一致性问题 |
| **如何优化查询性能？** | 创建合适的索引；优化查询语句；使用聚合管道优化复杂查询 |
| **如何管理权限？** | 设置合理的权限控制；使用角色管理权限；限制数据库访问 |
| **如何解决编码问题？** | 注意字符编码配置（UTF-8等）；统一字符集设置 |
| **如何保证高可用性？** | 配置副本集实现高可用；实现自动故障转移；定期数据备份 |

| 要点和思考方向 |
|----------------|
| MongoDB适合灵活的数据结构，不需要严格的表结构 |
| 需要关注集群配置、权限管理、编码设置等运维事项 |
| 适合游戏中的日志存储、玩家数据等场景 |
| 配置合适的集群模式（副本集、分片集群） |
| 优化查询性能和索引 |
| 注意数据一致性和高可用性配置 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「MongoDB」「副本集」「分片」「索引」「UTF-8」 |
| **方法** | 让 AI 设计集群配置、索引优化、权限与编码 |
| **应用** | 日志存储、玩家行为、灵活结构 |
| **提示词范例** | 「游戏日志、玩家行为数据用 MongoDB，请设计副本集与分片集群配置、索引与查询优化，以及权限管理与 UTF-8 编码设置」 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决性能瓶颈？** | 配置主从复制、读写分离；优化SQL查询和索引；使用连接池优化连接 |
| **如何保证数据一致性？** | 使用集群模式（主从、主主等）；配置数据同步机制；使用事务保证一致性 |
| **如何管理权限？** | 设置合理的权限控制；使用角色管理权限；限制数据库访问 |
| **如何解决编码问题？** | 注意字符编码配置；统一字符集设置 |
| **如何保证高可用性？** | 配置主从复制实现高可用；实现自动故障转移；定期数据备份 |

| 要点和思考方向 |
|----------------|
| MySQL适合需要ACID特性、强一致性的核心业务数据 |
| 需要关注集群配置、权限管理、编码设置、参数调优等运维事项 |
| 是游戏服务器最常用的关系型数据库 |
| 配置主从复制、读写分离提升性能 |
| 优化SQL查询和索引 |
| 注意数据一致性和高可用性配置 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「MySQL」「主从」「读写分离」「ACID」「连接池」 |
| **方法** | 让 AI 设计主从配置、读写分离、SQL 优化 |
| **应用** | 核心数据、账号角色、事务业务 |
| **提示词范例** | 「游戏核心数据用 MySQL，需 ACID 与强一致性，请设计主从复制与读写分离配置、连接池与 SQL 索引优化，以及高可用与故障转移方案」 |

## MSSQL

| 维度 | 内容 |
|------|------|
| **类型** | 关系数据库（SQL） |
| **作用** | 提供ACID特性、强一致性的数据存储，主要运行在Windows平台 |
| **应用场景** | Windows环境下的数据库选择；适合.NET技术栈的游戏服务器；需要Windows平台支持的业务；企业级数据库需求 |

| 问题 | 解决方向 |
|------|----------|
| **如何选择部署环境？** | 主要运行在Windows平台；考虑云服务支持；考虑容器化部署 |
| **如何与.NET技术栈集成？** | 使用Entity Framework等ORM框架；优化数据库连接 |
| **如何保证高可用性？** | 配置AlwaysOn可用性组；使用数据库镜像；实现自动故障转移 |
| **如何优化性能？** | 优化索引和查询；调整数据库参数；使用执行计划分析 |

| 要点和思考方向 |
|----------------|
| Microsoft SQL Server，主要运行在Windows平台 |
| 适合使用.NET技术栈的游戏项目 |
| 功能与MySQL类似，但生态和部署环境不同 |
| 配置AlwaysOn可用性组实现高可用 |
| 优化索引和查询性能 |
| 注意Windows平台特性和.NET技术栈集成 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「MSSQL」「AlwaysOn」「.NET」「Windows」 |
| **方法** | 让 AI 设计 MSSQL 高可用、.NET 集成、性能优化 |
| **应用** | Windows 环境、.NET 技术栈、企业级数据库 |
| **提示词范例** | 「.NET 游戏服务端用 MSSQL，请设计 AlwaysOn 可用性组高可用配置、Entity Framework 集成要点，以及索引与执行计划优化」 |

## 通用事项

| 维度 | 内容 |
|------|------|
| **作用** | 数据存储运维中的通用事项 |
| **应用场景** | 数据迁移；备份/恢复；日志查询；数据统计分析；数据回档 |

| 具体内容 | 说明 |
|----------|------|
| **数据迁移** | 云内数据迁移；跨境数据迁移；跨云数据迁移 |
| **备份/恢复** | 定期备份数据，确保数据安全，支持快速恢复 |
| **日志查询** | 查询数据库操作日志，用于问题排查和审计 |
| **数据统计分析** | 对数据库中的数据进行统计分析，支持运营决策 |
| **数据回档** | 归档至任意时间点，支持数据恢复和问题排查 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决数据迁移问题？** 迁移挑战：数据量大、跨地域跨云网络问题、服务中断 | 制定合理的迁移计划；使用增量迁移、双写等方式减少中断；使用云服务商提供的数据迁移工具；考虑迁移时间窗口、数据一致性、服务可用性 |
| **如何解决备份问题？** 备份挑战：策略不当、存储成本高、恢复时间长 | 制定备份策略（全量、增量、定时备份）；定期测试恢复流程；使用云服务商提供的备份工具；优化备份存储 |
| **如何使用日志查询？** | 查询数据库操作日志；用于问题排查和审计；分析性能问题；监控数据库操作 |
| **如何进行数据统计分析？** | 对数据库中的数据进行统计分析；支持运营决策；使用数据分析工具；生成统计报告 |
| **如何使用数据回档？** | 归档至任意时间点；支持数据恢复和问题排查；定期测试回档流程 |

| 要点和思考方向 |
|----------------|
| 数据迁移：需要考虑迁移时间窗口、数据一致性、服务可用性 |
| 备份/恢复：是数据安全的重要保障，需要定期验证备份的有效性 |
| 日志查询：有助于问题排查和性能分析 |
| 数据统计分析：支持运营数据分析和决策 |
| 数据回档：支持恢复到任意时间点，是处理数据问题的最后手段 |
| 制定合理的迁移和备份策略 |
| 定期测试恢复和回档流程 |
| 使用云服务商提供的工具和服务 |

| 类型 | AI Coding 指南 |
|------|----------------|

| **方法** | 让 AI 设计迁移方案、备份策略、恢复流程 |
