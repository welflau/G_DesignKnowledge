## 使用场景

| 场景 | 说明 |
|------|------|
| **Unity 脚本** | Unity 引擎的主要脚本语言之一 |
| **工具开发** | 命令行、图形界面工具，dotnet 框架提供高效开发环境 |

## 类

| 维度 | 内容 |
|------|------|
| **作用** | 面向对象基本单元，封装数据和行为 |
| **特性** | 封装、继承、多态（虚方法、抽象方法） |

| 问题 | 解决方向 |
|------|----------|
| **封闭类（sealed）扩展**：需扩展已封闭的类 | 接口注入；扩展方法 + 静态字典；组合；装饰器模式 |
| **类设计不当**：难以维护和扩展 | 单一职责；合理继承与组合；接口定义契约；避免过深继承 |

| 要点和思考方向 |
|----------------|
| 使用 sealed 防止不必要继承；用扩展方法扩展封闭类；遵循面向对象设计原则 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需扩展的类是否 sealed、扩展目的；如「第三方库的 sealed 类需要添加日志」「需在不修改源码的情况下扩展行为」 |
| **方法** | 提供类结构和约束；要求 AI 优先扩展方法、组合；避免反射或 IL 注入等侵入方式 |
| **应用** | 为 sealed 类生成扩展方法；重构继承为组合；生成接口+实现的设计 |
| **提示词范例** | 「这个 sealed 的 ConfigManager 类需要添加缓存逻辑，请用扩展方法实现，不修改原类」 |

## 虚方法

| 维度 | 内容 |
|------|------|
| **作用** | 实现多态，子类重写父类方法 |
| **关键字** | virtual（子类可不同实现）；override（重写）；base（调用父类实现） |

| 问题 | 解决方向 |
|------|----------|
| **性能开销**：虚函数表查找 | 性能关键路径减少调用；sealed 标记不需重写的方法；考虑接口替代 |
| **设计不当**：过度使用导致复杂 | 只在需多态处使用；接口定义契约；考虑组合替代继承 |

| 要点和思考方向 |
|----------------|
| 理解 virtual、override、base；注意性能开销；合理使用，避免过度设计 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明多态需求、是否热路径、子类数量；如「角色基类需不同攻击实现」「Update 中每帧调用」 |
| **方法** | 热路径要求 sealed override；提供继承层次；考虑接口 vs 虚方法选择 |
| **应用** | 将虚方法改为接口；为热路径方法添加 sealed；生成多态类层次 |
| **提示词范例** | 「这个 Enemy 基类的 TakeDamage 在战斗循环中调用，请评估是否应加 sealed 并说明理由」 |

## 抽象方法

| 维度 | 内容 |
|------|------|
| **作用** | 定义方法签名，强制子类实现，实现接口契约 |
| **特点** | abstract 不提供实现；含抽象方法的类必须是抽象类；子类必须实现所有抽象方法 |

| 问题 | 解决方向 |
|------|----------|
| **设计过度抽象** | 只在需强制实现处使用；考虑接口替代；平衡抽象与具体 |
| **抽象类实例化**：不能直接实例化 | 通过子类实例化；使用接口或具体类替代 |

| 要点和思考方向 |
|----------------|
| 抽象类不能直接实例化；合理使用，避免过度抽象；考虑接口替代 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需强制实现的契约、是否需默认实现；如「所有技能子类必须实现 Execute」「部分方法有默认实现」 |
| **方法** | 纯契约用接口；需共享实现用抽象类；要求 AI 说明选择理由 |
| **应用** | 将抽象类改为接口；生成抽象基类骨架；评估接口 vs 抽象类 |
| **提示词范例** | 「为技能系统设计基类，需要子类实现 Execute，但共享 Cooldown 逻辑，请用抽象类实现」 |

## 委托

| 维度 | 内容 |
|------|------|
| **作用** | 类，可定义函数指针，通过调用委托调用所指向的函数 |
| **应用场景** | 回调函数；参数传递；事件处理；策略模式 |

| 问题 | 解决方向 |
|------|----------|
| **委托为空**：导致 NullReferenceException | 调用前检查；`?.Invoke()` 安全调用；`??` 提供默认值 |
| **内存泄漏**：委托持有引用导致无法 GC | 及时取消订阅（`-=`）；弱事件模式；避免持有大对象 |
| **多播委托异常**：某方法异常影响后续 | 方法内部处理异常；手动遍历 `GetInvocationList()` 单独处理 |

| 要点和思考方向 |
|----------------|
| 使用安全调用；及时取消订阅；理解多播委托执行机制 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明回调场景、生命周期、是否可能为空；如「UI 按钮点击回调」「对象销毁后可能仍被调用」 |
| **方法** | 要求 `?.Invoke()` 或 null 检查；说明订阅方生命周期；避免闭包捕获大对象 |
| **应用** | 将委托改为安全调用；检查委托泄漏；生成带取消订阅的封装 |
| **提示词范例** | 「将这段委托调用改为安全调用，并添加订阅者在 View 销毁时自动取消的逻辑」 |

## 事件

| 维度 | 内容 |
|------|------|
| **作用** | 特殊的委托，向其他对象或组件发送通知 |
| **应用场景** | 消息传递；UI 事件；观察者模式；解耦依赖 |
| **特点** | 多播委托；触发时依次调用；只能在定义类内部触发 |

| 问题 | 解决方向 |
|------|----------|
| **内存泄漏**：未取消订阅导致无法 GC | 及时取消订阅；对象销毁时取消所有；弱事件；IDisposable |
| **触发顺序**：执行顺序不确定 | 不依赖顺序；需顺序时用列表/队列；`GetInvocationList()` 控制 |
| **异常处理**：某订阅者异常影响其他 | 内部捕获异常；手动遍历单独处理 |

| 要点和思考方向 |
|----------------|
| 及时取消订阅；不依赖执行顺序；妥善处理异常 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明事件用途、订阅者数量、是否跨 UI/逻辑层；如「背包变化通知多个 UI」「订阅者可能抛异常」 |
| **方法** | 要求取消订阅机制；不依赖调用顺序；异常隔离处理 |
| **应用** | 生成弱事件封装；检查事件泄漏；将事件改为带异常处理的发布订阅 |
| **提示词范例** | 「为 Inventory 的 OnItemChanged 事件添加订阅管理，确保 UI 销毁时自动取消，某订阅者异常不影响其他」 |

| **可读性差** | 封装为辅助方法；添加注释；代码生成替代 |

| 要点和思考方向 |
|----------------|
| 缓存反射结果；考虑表达式树、代码生成等替代方案 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明反射用途、调用频率、目标平台；如「配置表动态创建类型」「Unity IL2CPP 不支持部分反射」 |
| **方法** | 要求缓存 Type/MethodInfo；提供 IL2CPP 约束；考虑表达式树或代码生成替代 |
| **应用** | 将反射改为缓存委托；生成反射封装类；评估表达式树替代方案 |
| **提示词范例** | 「这段每帧用反射调用方法的代码要上 IL2CPP，请改为缓存 MethodInfo 或委托，并说明 IL2CPP 兼容性」 |

## 垃圾回收（GC）

| 维度 | 内容 |
|------|------|
| **作用** | 自动管理内存，回收不再使用的对象 |
| **特点** | 分代回收（0/1/2）；非确定性回收时机 |

| 问题 | 解决方向 |
|------|----------|
| **GC 暂停**：导致卡顿 | 减少分配；对象池；避免大对象（>85KB）；使用结构体 |
| **内存泄漏**：引用导致无法回收 | 避免循环引用；及时取消订阅；WeakReference；IDisposable |
| **性能优化** | 预设定容器大小；ArrayPool；StringBuilder；慎用 GC.Collect、Finalize |

| 要点和思考方向 |
|----------------|
| 减少分配、对象池；避免泄漏；性能关键路径优化 GC 压力 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明分配热点、目标平台、卡顿容忍度；如「战斗技能每帧 new 对象」「移动端 60fps 要求」 |
| **方法** | 要求对象池、ArrayPool、结构体；避免大对象堆；提供 GC 分析数据 |
| **应用** | 将 new 改为对象池；生成 StringBuilder 复用代码；分析并优化 GC 分配 |
| **提示词范例** | 「这段技能释放逻辑每帧 new List，导致 GC 卡顿，请改为对象池或预分配，Unity 移动端」 |

| 要点和思考方向 |
|----------------|

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明协程用途、数量、生命周期；如「分帧加载 100 个资源」「MonoBehaviour 销毁时需停止」 |
| **方法** | 要求保存 Coroutine 引用便于 StopCoroutine；提供销毁时机；考虑 UniTask 替代 |
| **应用** | 将协程改为可取消封装；生成协程管理器；评估 UniTask 迁移 |
| **提示词范例** | 「为这个分帧加载协程添加生命周期管理，确保场景切换时自动停止，并支持取消」 |

## 装箱拆箱

| 维度 | 内容 |
|------|------|
| **作用** | 值类型与引用类型之间的转换 |
| **说明** | 装箱：值类型→object；拆箱：object→值类型 |
| **应用场景** | 参数不预定数值类型；容器含非特定类型；与 object 交互 |

| 问题 | 解决方向 |
|------|----------|
| **性能问题**：装箱拆箱有开销 | 泛型替代 object；List\<int\> 而非 List\<object\>；避免循环中装箱拆箱 |
| **类型安全**：拆箱类型不匹配导致 InvalidCastException | is/as 类型检查；泛型避免拆箱；确保类型匹配 |

| 要点和思考方向 |
|----------------|
| 用泛型替代 object；注意拆箱类型安全；性能关键路径避免 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明容器用途、是否在循环中、类型是否固定；如「技能参数用 object 存储」「每帧 GetComponent」 |
| **方法** | 要求 List\<T\> 替代 ArrayList；避免值类型装箱；提供热路径上下文 |
| **应用** | 将 object 容器改为泛型；检查拆箱类型安全；生成泛型包装 |
| **提示词范例** | 「将这段用 Dictionary\<string, object\> 存技能参数的代码改为泛型，避免装箱并保证类型安全」 |

## 异步编程

| 维度 | 内容 |
|------|------|
| **作用** | 异步编程模型，避免阻塞主线程 |
| **应用场景** | IO 操作：网络、文件、数据库、Web API |
| **关键字** | Async/Await |

| 问题 | 解决方向 |
|------|----------|
| **死锁风险** | 避免 `.Result`、`.Wait()`；`ConfigureAwait(false)`；保持异步链；`Task.Run` |
| **性能考虑**：Unity 中协程 vs async/await | 协程通常性能更好；async/await 适合 IO；UniTask 在 Unity 中更好 |
| **异常处理** | try-catch；异常在 await 处抛出；`Task.Exception`；全局异常处理 |
| **取消操作** | `CancellationToken`；检查取消标记；`CancellationTokenSource` |

| 要点和思考方向 |
|----------------|
| 避免死锁；Unity 根据场景选择；妥善处理异常；支持取消 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明运行环境（Unity/ASP.NET）、IO 类型、取消需求；如「Unity 中加载 AssetBundle」「需要支持取消」 |
| **方法** | Unity 推荐 UniTask；避免 .Result/.Wait()；要求 CancellationToken；说明 ConfigureAwait |
| **应用** | 将同步 IO 改为 async/await；添加取消支持；Unity 中迁移到 UniTask |
| **提示词范例** | 「将这段 Unity 中同步加载资源的代码改为 UniTask 异步，支持取消，并处理加载失败」 |

## 异常

| 要点和思考方向 |
|----------------|
| 避免过度捕获；异常用于异常情况；using 确保资源释放 |

| 类型 | AI Coding 指南 |
|------|----------------|

| **方法** | 要求具体异常类型；避免 catch(Exception)；using 或 try-finally 释放资源 |
| **应用** | 将宽泛 catch 改为具体类型；添加资源释放；生成异常处理模板 |
| **提示词范例** | 「将这段 catch(Exception) 改为只捕获可处理的异常类型，并确保 Stream 在异常时正确关闭」 |

## 中间件

**是什么？在哪用？**
- **作用**：提供额外的功能和工具，简化开发
- **应用场景**：Unity开发、性能优化、代码简化

### UniTask

| 维度 | 内容 |
|------|------|
| **作用** | 异步流程控制库 |
| **应用场景** | Unity 高性能异步编程，比传统 Task 更好 |
| **优势** | 零 GC 分配；更好 Unity 集成；协程与 async/await 混合 |

| 问题 | 解决方向 |
|------|----------|
| **学习成本** | 阅读文档和示例；逐步迁移；理解与 Task 区别 |

### .Net

| 维度 | 内容 |
|------|------|
| **作用** | 跨平台运行时和开发环境 |
| **运行时** | IL2CPP（IL→C++，性能好）；AOT；Mono |

| 问题 | 解决方向 |
|------|----------|
| **平台兼容性** | 多平台测试；注意 API 差异；条件编译 |
| **性能差异**：IL2CPP 性能好但编译慢 | 开发用 Mono，发布用 IL2CPP；增量编译 |
| **IL2CPP 限制**：不支持部分反射等 | 了解限制；避免不支持特性；代码生成替代反射 |
