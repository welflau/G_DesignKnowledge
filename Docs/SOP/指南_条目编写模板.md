# 条目编写模板

> 配套文档：品类知识库构建 SOP v2.0  
> 本模板解决：**拿到数据后，怎么写出一条合格的知识/素材条目**

---

## 一、知识条目编写总原则

```
一条好的知识条目 = 【标题】+ 公式/框架/分类 + 具体参数 + 设计原理
                  ≥80字，推荐150~400字
                  抽象 > 实例，公式 > 配表，维度 > 数值
```

**写之前问自己三个问题：**
1. 一个策划读完这条，能直接用于设计吗？ → 不能就不够具体
2. 换一个同品类游戏还适用吗？ → 不适用就太具体了（应放素材表）
3. 有公式/框架/分类吗？ → 没有就是在讲故事而非传递知识

## 二、示例 1: 数值类条目（从配表到知识）

### 原始数据（Hades boon 配表 JSON）

```json
{
  "id": "boon_hearthbreak_strike",
  "god": ["god_aphrodite"],
  "name": "Heartbreak Strike",
  "type": ["attack"],
  "status_effect": ["weak"],
  "rarity": {
    "common": { "type": "+%", "value": 50 },
    "rare": { "type": "*", "value": { "min": 1.3, "max": 1.5 } },
    "epic": { "type": "*", "value": { "min": 1.8, "max": 2.0 } },
    "heroic": { "type": "*", "value": { "min": 2.3, "max": 2.5 } }
  }
}
```

### Step-by-Step 编写过程

**Step 1: 提炼公式**
- Common = base（+50%）
- Rare = Common × 1.3~1.5
- Epic = Common × 1.8~2.0
- Heroic = Common × 2.3~2.5
- 规律：每级是 Common 的倍数，乘法递增

**Step 2: 扩展为通用框架**
- 这不仅是 Aphrodite 的规律——所有神的 boon 都遵循相同的稀有度乘数结构
- 提炼出"肉鸽祝福稀有度体系"的通用模型

**Step 3: 加入设计原理**
- 为什么用乘法而非加法？→ 乘法使高稀有度 boon 在后期Build中价值指数增长
- Duo Boon 需要前置条件矩阵 → 创造"追求特定组合"的策略深度

**Step 4: 写成知识条目**

```
知识类型: 数值
描述: 【肉鸽祝福/Boon稀有度与数值膨胀体系】Hades boon稀有度乘数模型：
Common=base(如+50%伤害)/Rare=base×1.3~1.5/Epic=base×1.8~2.0/
Heroic=base×2.3~2.5，每级是Common的倍数而非固定加值（乘法膨胀）。
Duo Boon需前置条件矩阵（两个神各至少1个boon），提供独特强力效果。
这种乘法稀有度设计的目的：高稀有度boon在后期Build中价值指数增长，
驱动玩家追求"升级已有boon"而非"获取更多新boon"的决策。
来源: hades_json_data/patches/52/boon_data_aphrodite.js
标签: boon,稀有度,乘法膨胀,Duo Boon,Hades
质量: 深度
```

### 检查清单
- [x] 有【】标题
- [x] 有公式（稀有度乘数）
- [x] 有框架（4级稀有度体系）
- [x] 有设计原理（为什么用乘法）
- [x] ≥80字（实际约200字）
- [x] 标注来源
- [x] 3~6个标签

## 三、示例 2: 系统类条目（从文章到知识）

### 原始数据（spireofchaos.com 杀戮尖塔深度解析文章）

文章核心观点：
- 三层系统架构（战斗+地图 / 卡组+状态+商店 / 遗物+药水）
- 格挡回合末清零、能量限制3点、敌人意图可见
- 卡牌价值锚点：1能量≈6伤害

### Step-by-Step

**Step 1: 提炼系统架构**
- 一级系统（骨架）= 战斗 + 地图
- 二级系统（血肉）= 卡组 + 状态 + 商店
- 三级系统（灵魂）= 遗物 + 药水

**Step 2: 提炼设计原则**
- 三个反直觉设计及其目的

**Step 3: 补充数据锚点**
- 361卡/181遗物/4角色/66怪物 → 让条目有数据支撑

**Step 4: 写成知识条目**

```
知识类型: 策划
描述: 【竞品·杀戮尖塔核心系统拆解】三层系统架构：一级(骨架)=战斗系统+地图系统，
二级(血肉)=卡组+状态+商店，三级(灵魂)=遗物+药水。核心数据：361卡/181遗物/
42药水/66怪物/52事件/4角色。反直觉三设计：格挡回合末清零(禁止龟缩)/
能量限3(资源匮乏=策略深度)/敌人意图可见(降低门槛拉高上限)。
卡牌价值锚点：1能量≈6伤害。Steam 97%好评/全球销量数百万份。
来源: spireofchaos.com杀戮尖塔深度解析+spire-archive数据库
标签: 杀戮尖塔,三层架构,Ascension,97%好评
质量: 深度
```

## 四、示例 3: 竞品拆解类条目（从多来源到知识）

### 原始数据（多来源综合）

- GameRes 文章：无尽冬日产品模型解析
- SensorTower 数据：月流水/全球排名
- 网络搜索：核心循环/商业化设计

### Step-by-Step

**Step 1: 确定拆解维度（每款竞品固定3条）**

| 条目# | 维度 | 覆盖内容 |
|--------|------|---------|
| 1 | 核心循环与系统 | 核心玩法是什么+与同品类差异化在哪 |
| 2 | 商业化与活动 | 怎么赚钱+付费深度+活动设计 |
| 3 | 社交/留存/特色 | 社交设计+留存机制+该游戏最独特的设计 |

**Step 2: 每条聚焦一个维度，不要混杂**

**Step 3: 必须包含具体数据**
- 竞品拆解如果没有具体数字就是废话
- 必须有：用户量/收入/具体参数/具体机制名称

**Step 4: 写成竞品拆解条目**

```
知识类型: 策划
描述: 【竞品·无尽冬日核心循环与融合设计拆解】SLG+模拟经营+放置三层融合，
全球累计流水超300亿人民币。核心循环：模拟经营产出资源→资源投入英雄养成
与建筑升级→养成结果在放置战斗/SLG大地图中验证→逐步导向联盟GvG消耗。
核心创新：前期完全不像SLG(D1~3只有经营+放置)→大熔炉Lv10解锁野外SLG。
双战斗系统：放置(5英雄自走棋)+SLG大地图(采集/集结/攻城)。
地图分层：荒原(PVE)→中间(联盟扩张)→太阳城(核心PVP)。
来源: GameRes产品分析+SensorTower
标签: 无尽冬日,三层融合,大熔炉,双战斗系统
质量: 深度
```

## 五、素材条目编写（与知识条目相反）

素材表**鼓励保留具体数据**。写法很简单：

```
知识类型: 数值
描述: 【杀戮尖塔完整数据统计表】STS1: 卡牌361张(Ironclad约78/Silent约75/
Defect约75/Watcher约75/Colorless约20/Curse约15), 遗物181个
(Common约60/Uncommon约50/Rare约30/Boss约25/Shop约10/Event约6/Starter4),
药水42个, 怪物66个, 事件52个, 效果146个, 角色4个。
商店定价: Common卡50~75金/Uncommon卡150~225金/Rare卡300~450金/
移除卡75金(递增25)。
来源: spire-archive
标签: 杀戮尖塔,卡牌统计,遗物统计,商店定价
质量: 深度
```

**素材表要点：保留完整数字、保留完整列表、保留配表明细。**

## 六、常见错误

| 错误 | 示例 | 修复 |
|------|------|------|
| 只有观点没有数据 | "杀戮尖塔的战斗系统设计得很好" | 加上具体参数：361卡/181遗物/1能量≈6伤害 |
| 只有数据没有解释 | "Common=60%, Rare=10%" | 加上设计原理：为什么这样分配概率 |
| 太短(流水账) | "杀戮尖塔有4个角色，每个不同。" | 展开说明每个角色的独占资源和定位差异 |
| 太长(论文级) | 2000字描述一个系统 | 压缩到300~400字，把具体数据移入素材表 |
| 知识表混入配表 | 逐一列举每个角色的属性值 | 属性值放素材表，知识表只写"属性维度+分类方法" |
