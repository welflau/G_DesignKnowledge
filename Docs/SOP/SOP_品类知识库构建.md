# 品类知识库构建 SOP

> 版本：v2.0 | 2026-04-21  
> 基于 SLG 品类完整跑通 + 通用层补全的经验沉淀

---

## 一、SOP 概览

```
Step 1: 品类定义 → 在 knowledge_game_meta.csv 中新增一行
Step 2: 知识全景图 → 画出品类应覆盖的完整知识维度
Step 3: 数据源盘点 → 确认可用仓库和竞品资料
Step 4: 知识条目编写 → 按抽取规范生成 knowledge_xxx.csv
Step 5: 素材表提取 → 可选，生成 knowledge_xxx_material.csv
Step 6: 自审 → 用标准审查脚本验证
Step 7: 修复迭代 → 修复审查发现的问题
Step 8: 交付 → 上传 git + gnpc 平台
```

预计每个品类耗时：**2~4 小时**（有数据源的情况下）

---

## 二、Step 1: 品类定义

### 操作
在统一的 `knowledge_game_meta.csv` 中追加一行（**表头使用中文名**）：

```csv
游戏名,游戏名英文,游戏介绍,游戏类型,对应游戏知识库的表名,对应游戏素材知识库的表名,覆盖的知识维度,数据来源说明,最后更新时间
```

### 三层分类规则
| 层级 | 游戏名示例 | 说明 |
|------|-----------|------|
| 第1层-通用 | 通用 | 跨品类方法论（game_name_en=common） |
| 第2层-品类 | SLG / 肉鸽 | 品类通用设计范式 |
| 第3层-游戏 | 明日方舟 / 杀戮尖塔 | 具体游戏数据和案例 |

### 检查清单
- [ ] `游戏名英文` 小写字母开头，仅含小写字母/数字/下划线，≤50字符
- [ ] `游戏介绍` 50~200 字
- [ ] `游戏类型` 从枚举中选择（可多选逗号分隔）
- [ ] `对应游戏知识库的表名` = `knowledge_` + 游戏名英文（**不可自定义**）
- [ ] `对应游戏素材知识库的表名` = `knowledge_` + 游戏名英文 + `_material`（有素材库时必填）
- [ ] `覆盖的知识维度` 取值与知识类型枚举一致
- [ ] `最后更新时间` 格式 YYYY-MM-DD

### 游戏类型枚举
```
通用、SLG、RTS、塔防RPG、开放世界、回合制RPG、
弹幕RPG、桌游、肉鸽、叙事、MOBA、卡牌
```

## 三、Step 2: 知识全景图

### 目的
画出品类应覆盖的**完整知识维度**，作为编写和缺口检查的清单。

### 模板结构
```
[品类名] 知识全景图
├── 数值体系
│   ├── 战斗公式（核心伤害模型）
│   ├── 成长曲线（经验/属性/等级）
│   ├── 经济系统（资源/产出/消耗/通胀）
│   ├── 概率系统（抽卡/掉落/暴击）
│   └── 平衡方法论
├── 系统设计
│   ├── 核心系统 A（品类特有）
│   ├── 核心系统 B（品类特有）
│   ├── ...
│   └── AI 系统
├── 玩法设计
│   ├── 核心循环
│   ├── 子品类对比
│   ├── 留存模型
│   └── 社交设计
├── 策划方法论
│   ├── 数值选型指南
│   ├── 竞品分析（头部3~5款深度拆解）
│   └── 商业化设计
├── 叙事设计
│   ├── 叙事结构
│   ├── 角色塑造
│   └── 世界观构建
└── 商业化
    ├── 付费体系
    ├── LTV 模型
    └── 运营节奏
```

### 要求
- 每个叶节点对应知识表中的 **1 条知识条目**
- 标注每个节点的状态：✅已有 / 🔴缺失 / ⚠️薄弱
- 最终全景图中**缺失项应为 0**

### 竞品拆解标准
| 层级 | 数量 | 拆到什么程度 |
|------|------|------------|
| 头部竞品深度拆解 | **3~5款** | 系统级：每款拆出3条知识（战斗/系统/商业化） |
| 配表级数据源 | **2~3款** | 有源码/JSON可提取：每款拆出5~8条素材 |
| 提及级参考 | 3~5款 | 仅在对比中引用 |

## 四、Step 3: 数据源盘点

### 操作
检查可用的数据来源，评估数据质量：

| 来源类型 | 适用 | 标注 |
|---------|------|------|
| 有完整配表 JSON/YAML/CSV | 知识表(抽象) + 素材表(明细) | 仓库路径 |
| 有设计文档/README | 知识表(系统架构/玩法设计) | 仓库路径 |
| 有代码实现 | 提炼算法和公式入知识表 | 仓库路径 |
| 无对应数据源 | 从网络搜索/竞品分析补充 | "网络搜索综合" |
| 知乎/GameRes等深度文章 | 竞品拆解/实测验证数据 | URL |

## 五、Step 4: 知识条目编写

### CSV 表头（使用中文名）
```csv
知识类型,描述,来源,标签,质量
```

### 知识类型枚举
| 类型 | 说明 |
|------|------|
| 玩法 | 玩法机制、核心循环、品类设计 |
| 数值 | 公式体系、配表规律、平衡方法 |
| 系统 | 系统架构、模块设计、流程设计 |
| 剧情 | 叙事结构、角色设计、文案方法（允许保留具体实例） |
| 综合 | 跨维度知识 |
| 策划 | 方法论、竞品分析、商业化 |

### 抽取原则（知识表专用）
1. **抽象优于实例**：描述设计规律和公式，不逐一列举
2. **维度优于数值**：给出属性维度体系和分类方法
3. **公式优于配表**：提炼计算公式，不粘贴整张配表
4. **量化边界优于枚举**：用阈值/区间定义类型边界

### 条目质量标准
- 每条 **≥80 字**（深度级推荐 150~400 字）
- 标题使用 **【】** 标记
- 必须包含 **公式/框架/分类** 中的至少一种
- tags **3~6 个**，逗号分隔
- 质量等级：深度(>200字含完整公式) / 实质(100~200字) / 中等 / 轻量 / 索引

### 类型分布目标
| 类型 | 占比 | 说明 |
|------|------|------|
| 数值 | 25~35% | 品类核心公式体系 |
| 系统 | 20~30% | 核心系统设计 |
| 策划 | 15~25% | 方法论+商业化+竞品 |
| 玩法 | 15~20% | 品类玩法设计范式 |
| 剧情 | 5~15% | 叙事设计（按品类调整） |
| 综合 | 0~5% | 可选 |

**单一类型不超过 50%。**

## 六、Step 5: 素材表提取

### 何时需要素材表
- 品类有丰富的开源配表数据
- 知识表中的抽象规则需要具体数据支撑
- **不是所有品类都需要素材表**

### 素材表原则（与知识表相反）
1. **保留完整配表**：角色全属性表、Boss HP 明细
2. **保留具体数值**：每个角色的初始属性值
3. **保留枚举明细**：全部道具列表、全部技能列表
4. **标题加【】标记**

### CSV 表头（与知识表一致，使用中文名）
```csv
知识类型,描述,来源,标签,质量
```

## 七、Step 6: 自审

### 标准审查项

**Meta 表（在 knowledge_game_meta.csv 中检查对应行）：**
1. 游戏介绍 50~200 字
2. 游戏名英文 格式合规（小写+下划线）
3. 对应游戏知识库的表名 = knowledge_ + 游戏名英文
4. 对应游戏素材知识库的表名 = knowledge_ + 游戏名英文 + _material（有素材时）
5. 游戏类型 在枚举内
6. 最后更新时间 YYYY-MM-DD
7. 覆盖的知识维度 取值与知识类型枚举一致

**知识表/素材表：**
8. 知识类型 全部有值
9. 描述 全部有值
10. 知识类型 在枚举内
11. 质量 在枚举内
12. 【】标题齐全
13. 标题无重复
14. 无过短条目（知识表≥80字/素材表≥100字）
15. CSV 行数正确（无换行污染）
16. 类型分布均衡（单一类型 <50%）

**交叉检查：**
17. 新品类知识表 ∩ 通用层(common) 无重复标题
18. 素材表含具体数值/配表数据

### 通过标准
- **全部通过**方可交付
- 任何 🔴 项必须修复

## 八、Step 7: 修复迭代

### 常见问题及修复方法

| 问题 | 原因 | 修复 |
|------|------|------|
| 条目过短 | 为扩充数量压缩了内容 | 恢复完整公式/框架，宁可少条目不可低质量 |
| 类型偏科 | 数据源偏向某一类型 | 从网络搜索/竞品分析补充弱势类型 |
| CSV 换行污染 | 描述含未转义的换行 | 用 Python csv 模块写入，自动处理转义 |
| 标题重复 | 同一主题写了多条 | 合并为一条深度条目 |
| 混入具体数据 | 知识表和素材表职责混淆 | 具体配表移入素材表，知识表只保留抽象规则 |
| 与通用层重复 | 跨品类通用内容放错了位置 | 迁移到 knowledge_common.csv |

### 核心原则
> **宁可 50 条高质量，不要 100 条注水。知识表的价值在内容深度，不在数量。**

# 附录 A: 知识库表结构设计

> 以下为知识库的底层数据结构规范，SOP 中所有操作均基于此结构。

---

## A.1 整体架构

知识库分为 **meta表**（索引层）、**游戏知识表**（内容层）和 **游戏原始素材知识库表**（素材层）三层结构。
- 最终存储在 gnpc 平台（空间：gflow）
- 中间产物缓存在 git：https://git.woa.com/gflow/gflow-knowledge/
- 所有字段统一为 string 类型，多值字段用英文逗号（,）分隔
- 日期格式统一为 YYYY-MM-DD（如 2026-04-16）
- **生成 excel 时表头统一使用中文名**（每张表的"字段中文名"列即为 excel 表头）
- Meta 表 code（表名）：**knowledge_game_meta**
- 知识库表名命名规则：**knowledge_** 前缀 + **game_name_en**，不允许自定义（如 game_name_en="arknights" → knowledge_table_name="knowledge_arknights"）
- 素材库表名命名规则：**knowledge_** 前缀 + **game_name_en** + **_material** 后缀（如 game_name_en="arknights" → material_table_name="knowledge_arknights_material"）

### 知识表与素材表的定位区别

| 维度 | 游戏知识表（内容层） | 游戏原始素材知识库表（素材层） |
|------|---------------------|-------------------------------|
| 定位 | 规则层+锚点层：设计思想、公式体系、分类框架 | 素材层：原始配表、明细数据、具体实例 |
| 抽象程度 | 高度抽象，提炼规律和公式 | 保留原始数据粒度，含具体数值明细 |
| 典型内容 | 伤害公式、属性维度体系、系统架构、设计哲学 | 角色属性配表、Boss HP明细、道具列表、价格表 |
| 使用场景 | 设计参考、方法论检索、跨游戏对比 | 数据查询、配表参考、具体数值验证 |

### 知识分层检索逻辑

知识按三层组织，检索时逐层细化：

1. **通用知识**：跨品类/跨游戏的通用方法论和模板（meta表中 game_name="通用"）
2. **品类通用知识**：某品类的通用设计范式（meta表中 game_name="肉鸽"/"SLG" 等品类名）
3. **具体游戏知识**：特定游戏的实际数据和案例（meta表中 game_name="明日方舟" 等具体游戏名）

| quality_level | 质量 | ❌ | 内容质量层级（深度/实质/中等/轻量/索引） |

### 知识类型枚举

| 知识类型 | 说明 | 对应仓库目录 |
|---------|------|-------------|

## A.4 游戏原始素材知识库表（素材表）

类型：数据库型，每个游戏可选配一个素材库，主 key 为 material_id（由系统自动生成，无需手动填写）。所有字段均为 string 类型。

表名命名规则：**knowledge_** + **game_name_en** + **_material**（如 game_name_en="binding_of_isaac" → "knowledge_binding_of_isaac_material"）

| 字段名 | 字段中文名 | 必填 | 说明 |
|--------|-----------|------|------|
| knowledge_type | 知识类型 | ✅ | 知识类型（与知识表共用枚举） |
| knowledge_desc | 描述 | ✅ | 原始素材数据内容（配表明细、具体数值列表等） |
| source_path | 来源 | ❌ | 原始数据路径/来源URL |
| tags | 标签 | ❌ | 标签，多值用逗号分隔（用于检索） |
| quality_level | 质量 | ❌ | 内容质量层级（深度/实质/中等/轻量/索引） |

### 素材表与知识表的字段一致性

素材表与游戏知识表**共享相同的字段结构**，区别在于内容粒度：
- **知识表**：knowledge_desc 中存放高度抽象的规则、公式、框架
- **素材表**：knowledge_desc 中存放原始配表数据、具体数值明细、完整列表

### 素材表内容规范

素材表**允许且鼓励**保留具体数据实例，与知识表的抽取原则相反：

1. **保留完整配表**：如角色全属性表、Boss HP明细、道具完整列表
2. **保留具体数值**：如每个角色的初始属性值、每个道具的具体效果参数
3. **保留枚举明细**：如全部变身触发条件、全部道具池归属关系
4. **知识条目标题建议加【】标记**：如"【角色初始属性配表】"、"【Boss HP配表】"，便于快速识别素材类型

### 素材表适用场景

- 游戏有丰富的开源配表数据可沉淀时
- 需要保留原始数值明细供查询参考时
- 知识表中的抽象规则需要配套的具体数据支撑时
- 并非所有游戏都需要素材表，仅在有充足原始数据时创建

- 所有字段均为 string 类型，多值字段用英文逗号分隔

## A.6 游戏知识抽取方法

抽取知识时遵循以下原则，重点提炼**设计思想、公式体系、分类框架**，而非罗列具体数据实例。

### 抽取原则

1. **抽象优于实例**：描述设计规律和公式，不逐一列举每个具体对象的属性值
2. **维度优于数值**：给出属性维度体系和分类方法，而非具体数字
3. **公式优于配表**：提炼计算公式（含多乘区拆解），而非粘贴整张配表
4. **量化边界优于枚举**：用阈值/区间定义类型边界（如"肉盾：防御力>X"），而非列举所有单位

### 各知识类型的抽取规范

**数值类知识**：
- 提炼公式体系：如伤害 = 基础攻击 × 技能倍率 × 属性克制 × 暴击乘区 × 减防乘区
- 给出属性维度列表：如怪物具备{生命/攻击/防御/速度/抗性}等维度
- 给出分类定义及量化标准：如怪物分为{肉盾/输出/辅助/精英/Boss}，肉盾定义为防御力>基准值×1.5且生命>基准值×2
- 描述成长曲线类型：如线性/指数/对数/阶梯，以及关键拐点

**系统类知识**：
- 描述系统架构和模块关系，不描述具体UI布局
- 给出系统间的数据流向和依赖关系
- 提炼设计模式和复用结构

**玩法类知识**：
- 描述核心循环和机制原理，不描述具体关卡细节
- 提炼玩法公式和概率模型
- 给出机制分类和组合规则

**剧情类知识**：
- 可保留具体文本内容（对话、故事、世界观设定等）
- 剧情类是唯一允许保留具体实例的知识类型

### 抽取示例

**错误示例**（逐一列举）：
```
哥布林：HP=100,ATK=15,DEF=5; 史莱姆：HP=80,ATK=10,DEF=3; 骷髅兵：HP=120,ATK=20,DEF=8 ...
```

**正确示例**（公式+维度+分类）：
```
怪物属性维度：生命值/攻击力/防御力/攻击速度/移动速度/属性抗性。
怪物分类：小怪(HP<基准×1)/精英(HP=基准×3~5)/Boss(HP=基准×10+)。
伤害公式：最终伤害 = ATK × 技能倍率 × (1-DEF/(DEF+常数)) × 属性克制系数 × 暴击乘区。
属性克制系数：克制=1.5/被克=0.5/无关=1.0。
```

## 附录 B: 审查脚本模板

```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
"""品类知识库 CSV 审查脚本 v2.0"""
import csv, re, sys

GAME = sys.argv[1] if len(sys.argv) > 1 else "slg"

checks = []
def check(name, ok, detail=""):
    checks.append((name, ok, detail))

# Meta（从统一meta表中检查对应行）
with open('knowledge_game_meta.csv', encoding='utf-8') as f:
    meta_rows = list(csv.DictReader(f))
meta = None
for r in meta_rows:
    if r['游戏名英文'] == GAME:
        meta = r
        break
if not meta:
    print(f"🔴 Meta表中找不到 game_name_en={GAME}")
    sys.exit(1)

check('meta.intro 50~200字', 50 <= len(meta['游戏介绍']) <= 200)
check('meta.en 格式', bool(re.match(r'^[a-z][a-z0-9_]{0,49}$', meta['游戏名英文'])))
check('meta.ktn', meta['对应游戏知识库的表名'] == f'knowledge_{GAME}')
mtn = meta.get('对应游戏素材知识库的表名', '')
check('meta.mtn', not mtn or mtn == f'knowledge_{GAME}_material')
valid_types = ['通用','SLG','RTS','塔防RPG','开放世界','回合制RPG',
               '弹幕RPG','桌游','肉鸽','叙事','MOBA','卡牌']
check('meta.type 枚举', all(t.strip() in valid_types
      for t in meta['游戏类型'].split(',')))
valid_dims = ['玩法','数值','系统','剧情','综合','策划']
check('meta.dims 枚举', all(d.strip() in valid_dims
      for d in meta['覆盖的知识维度'].split(',')))

# 知识表
with open(f'knowledge_{GAME}.csv', encoding='utf-8') as f:
    krows = list(csv.DictReader(f))
valid_kt = ['玩法','数值','系统','剧情','综合','策划']
valid_ql = ['深度','实质','中等','轻量','索引','']
check('k.类型有值', all(r['知识类型'] for r in krows))
check('k.描述有值', all(r['描述'] for r in krows))
check('k.类型枚举', all(r['知识类型'] in valid_kt for r in krows))
check('k.质量枚举', all(r.get('质量','') in valid_ql for r in krows))
check('k.标题【】', all(r['描述'].startswith('【') for r in krows))
titles = [re.match(r'【(.+?)】', r['描述']).group(1)
          for r in krows if re.match(r'【(.+?)】', r['描述'])]
check('k.标题唯一', len(titles) == len(set(titles)))
check('k.≥80字', all(len(r['描述']) >= 80 for r in krows))
with open(f'knowledge_{GAME}.csv', encoding='utf-8') as f:
    check('k.CSV完整', len(f.read().strip().split('\n')) == len(krows) + 1)
types = {}
for r in krows:
    types[r['知识类型']] = types.get(r['知识类型'], 0) + 1
check('k.均衡<50%', max(v / len(krows) for v in types.values()) < 0.5)

# 交叉检查(与common)
with open('knowledge_common.csv', encoding='utf-8') as f:
    c_rows = list(csv.DictReader(f))
c_titles = set(re.match(r'【(.+?)】', r['描述']).group(1)
               for r in c_rows if re.match(r'【(.+?)】', r['描述']))
overlap = set(titles) & c_titles
check('交叉:与common无重复', len(overlap) == 0)

# 汇总
passed = sum(1 for _, ok, _ in checks if ok)
for name, ok, detail in checks:
    print(f'{"✅" if ok else "🔴"} {name}'
          + (f' — {detail}' if detail and not ok else ''))
print(f'\n{passed}/{len(checks)}'
      f' {"🎉 全部通过" if passed == len(checks) else "需修复"}')
```
